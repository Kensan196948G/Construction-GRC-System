## 1. 目的・背景

### 1.1 背景

建設業界は複数の法令・規格への準拠が求められる規制産業であり、みらい建設工業では以下の多岐にわたる要件への対応が必要である。

**情報セキュリティ・IT系：**
- ISO27001:2022（情報セキュリティマネジメントシステム）- 全93管理策
- NIST CSF 2.0（サイバーセキュリティフレームワーク）- 6機能・22カテゴリ
- ISO20000-1:2018（ITサービスマネジメント）

**建設業法令：**
- 建設業法（許可・経営事項審査・下請契約）
- 公共工事の品質確保の促進に関する法律（品確法）
- 労働安全衛生法（現場安全管理）
- 下請代金支払遅延等防止法（下請法）

**現状の課題：**

| 課題 | 現状 | 目標 |
|------|------|------|
| コンプライアンス可視化 | 部門ごとに分散・属人的 | 全社横断的なリアルタイム可視化 |
| 監査対応工数 | 年間約500時間 | 年間200時間以下（60%削減） |
| リスクアセスメント頻度 | 年1回 | 四半期ごと＋随時（継続的モニタリング） |
| 法改正対応 | 手動確認（対応遅れリスク） | 自動トラッキング・アラート通知 |
| 適用宣言書（SoA）作成 | 手動Excel管理（3日） | 自動生成（即時） |
| 是正処置追跡 | メール・Excel管理 | ワークフロー管理・自動リマインダー |

### 1.2 目的

多法令・多規格に対応した統合GRC（Governance, Risk, Compliance）基盤を構築し、以下を実現する。

1. **ガバナンス強化**: 経営層がリスク・コンプライアンス状況をリアルタイムに把握できる体制の構築
2. **リスク管理の高度化**: リスクアセスメントの継続的実施と残存リスクの定量的モニタリング
3. **コンプライアンス効率化**: 多規格への準拠状況の一元管理と証跡収集の自動化
4. **監査対応の効率化**: 内部監査の計画から是正処置追跡までの一貫したワークフロー管理
5. **建設業特有要件への対応**: 現場安全・施工品質・下請管理等の業界固有リスクへの対応

### 1.3 期待される効果

| 効果指標 | 現状値 | 目標値 | 達成時期 |
|---------|--------|--------|---------|
| 監査対応工数 | 500時間/年 | 200時間/年 | 運用開始後6ヶ月 |
| リスクアセスメント頻度 | 1回/年 | 4回/年＋随時 | 運用開始時 |
| コンプライアンス可視化率 | 30% | 100% | 運用開始後3ヶ月 |
| SoA作成時間 | 3日 | 即時（自動生成） | 運用開始時 |
| 法改正検知リードタイム | 平均30日 | 7日以内 | 運用開始後3ヶ月 |
| 是正処置完了率 | 70% | 95%以上 | 運用開始後6ヶ月 |

### 3.2 組織スコープ

| 部門 | 主要役割 | 利用モジュール |
|------|---------|--------------|
| IT部門（主管） | システム管理・ISO27001/NIST CSF対応 | 全モジュール |
| 経営管理部 | 建設業法・下請法対応 | CMP, RPT |
| 工事部門 | 品確法・施工品質管理 | RSK, CMP |
| 安全管理部 | 労安法・現場安全管理 | RSK, CMP |

### 3.3 スコープ外

以下は本プロジェクトのスコープ外とする。

- 外部認証機関との直接連携（将来検討）
- 施工管理システム（BIM/CIM）との統合
- 経理・会計システムとの連携
- モバイルアプリ対応（将来フェーズで検討）
- 多言語対応（日本語のみ）

## 4. 用語定義

| 用語 | 定義 |
|------|------|
| GRC | Governance（統治）, Risk（リスク）, Compliance（法令遵守）の統合管理 |
| SoA | Statement of Applicability（適用宣言書）。ISO27001で要求される管理策の適用・除外を宣言する文書 |
| CAP | Corrective Action Plan（是正処置計画）。不適合に対する改善計画 |
| リスクヒートマップ | 発生可能性と影響度の2軸でリスクを可視化するマトリクス図 |
| リスクレジスター | 組織が特定した全リスクを一覧化した台帳 |
| RBAC | Role-Based Access Control（ロールベースアクセス制御） |
| 経審 | 経営事項審査。建設業法に基づく公共工事の入札参加資格審査 |
| 管理策 | ISO27001附属書Aに定義されるセキュリティ管理の具体的手段 |
| 証跡 | コンプライアンスの準拠を証明するための記録・文書・ログ等 |
| NIST CSF | National Institute of Standards and Technology Cybersecurity Framework |
| 残存リスク | リスク対応策を講じた後も残るリスク |
| 固有リスク | リスク対応策を講じる前のリスク |

```

  ├──→ 管理策管理（CTL）: リスクに対応する管理策の紐付け
  ├──→ コンプライアンス管理（CMP）: リスクに関連する法令要件の紐付け
  └──→ レポート（RPT）: リスクヒートマップ・トレンド分析

管理策管理（CTL）
  ├──→ コンプライアンス管理（CMP）: 管理策の証跡・準拠率
  └──→ レポート（RPT）: SoA生成・準拠率レポート

内部監査管理（AUD）
  ├──→ 管理策管理（CTL）: 監査所見と管理策の紐付け
  ├──→ コンプライアンス管理（CMP）: 監査所見と法令要件の紐付け
  └──→ レポート（RPT）: 監査報告書生成

レポート（RPT）
  └──→ 全モジュールのデータを集約して可視化
```

## 6. リスク管理要件（RSK）

### RSK-001: リスクレジスター管理

| 項目 | 内容 |
|------|------|
| **要件ID** | RSK-001 |
| **要件名** | リスクレジスター管理 |
| **優先度** | 必須 |
| **カテゴリ** | リスク管理 |

**概要:**
全社的なリスクの登録・分類・評価・対応計画・ステータス管理を行うリスクレジスター機能。

**詳細要件:**

1. リスクの新規登録・編集・削除（論理削除）
2. リスクID自動採番（カテゴリプレフィックス付き: RISK-IT-001, RISK-CON-001等）
3. リスクカテゴリ分類: IT/物理/法務/建設/環境/財務/人的
4. リスクソース分類: 脅威/脆弱性/プロセス/外部環境
5. 固有リスク評価（発生可能性1-5 x 影響度1-5の5段階評価）
6. 残存リスク評価（対策後の再評価）
7. リスクスコア自動計算（発生可能性 x 影響度）
8. リスクレベル自動判定: LOW(1-4)/MEDIUM(5-9)/HIGH(10-14)/CRITICAL(15-25)
9. リスク対応戦略の記録: 受容/軽減/移転/回避
10. リスクオーナーのアサイン
11. 対応計画・目標期日の設定
12. 関連するISO27001管理策との紐付け
13. 関連する法令要件との紐付け
14. リスクレビュー日の管理・リマインダー
15. リスクの検索・フィルタリング（カテゴリ/レベル/ステータス/オーナー）
16. リスク一覧のCSV/Excel出力
17. 変更履歴の全件記録（誰が・いつ・何を変更したか）

**データ要件:**

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| risk_id | VARCHAR(20) | Yes | 自動採番リスクID |
| title | VARCHAR(300) | Yes | リスク名称 |
| description | TEXT | Yes | リスクの説明 |
| category | VARCHAR(50) | Yes | リスクカテゴリ |
| source | VARCHAR(50) | No | リスクソース |
| likelihood_inherent | SMALLINT(1-5) | Yes | 固有リスク発生可能性 |
| impact_inherent | SMALLINT(1-5) | Yes | 固有リスク影響度 |
| risk_score_inherent | SMALLINT | Auto | 固有リスクスコア（自動計算） |
| likelihood_residual | SMALLINT(1-5) | No | 残存リスク発生可能性 |
| impact_residual | SMALLINT(1-5) | No | 残存リスク影響度 |
| risk_score_residual | SMALLINT | Auto | 残存リスクスコア（自動計算） |
| treatment_strategy | VARCHAR(20) | Yes | 対応戦略 |
| treatment_plan | TEXT | No | 対応計画詳細 |
| risk_owner_id | UUID | Yes | リスクオーナー |
| target_date | DATE | No | 対応目標日 |
| status | VARCHAR(20) | Yes | ステータス |
| review_date | DATE | No | 次回レビュー日 |

### RSK-002: リスクアセスメントワークフロー

| 項目 | 内容 |
|------|------|
| **要件ID** | RSK-002 |
| **要件名** | リスクアセスメントワークフロー |
| **優先度** | 必須 |
| **カテゴリ** | リスク管理 |

**概要:**
リスク評価会議の招集・実施・記録を一元管理するワークフロー機能。

**詳細要件:**

1. リスクアセスメント計画の作成（対象範囲・参加者・スケジュール）
2. アセスメント実施の記録（評価結果・議事録・合意事項）
3. ワークフローステータス管理: 計画中/実施中/レビュー中/完了/承認済
4. アセスメント結果に基づくリスクレジスター自動更新
5. 四半期ごとの定期アセスメントスケジューリング
6. 臨時アセスメントの随時起票
7. アセスメント参加者への通知・リマインダー（メール/システム内通知）
8. 過去のアセスメント履歴の参照・比較
9. 承認ワークフロー（リスクオーナー→GRC管理者→経営層）

### RSK-003: リスクヒートマップ

| 項目 | 内容 |
|------|------|
| **要件ID** | RSK-003 |
| **要件名** | リスクヒートマップ |
| **優先度** | 必須 |
| **カテゴリ** | リスク管理 |

**概要:**
発生可能性（Likelihood）x 影響度（Impact）の5x5マトリクスによるリスクの視覚的可視化。

**詳細要件:**

1. 5x5マトリクス形式のヒートマップ表示
2. 各セルにリスク件数・リスクIDの表示
3. リスクレベルに応じた色分け（GREEN/YELLOW/ORANGE/RED）
4. 固有リスクと残存リスクの切替表示
5. カテゴリ別フィルタリング
6. 部門別フィルタリング
7. セルクリックによるリスク詳細一覧の表示
8. ヒートマップ画像の出力（PNG/PDF）
9. 前回アセスメントとの比較表示（変動の可視化）
10. リアルタイム更新（リスクデータ変更時の自動反映）

### RSK-004: リスク受容・移転・回避管理

| 項目 | 内容 |
|------|------|
| **要件ID** | RSK-004 |
| **要件名** | リスク受容・移転・回避管理 |
| **優先度** | 必須 |
| **カテゴリ** | リスク管理 |

**概要:**
リスク対応戦略（受容・軽減・移転・回避）の記録と承認ワークフロー。

**詳細要件:**

1. 対応戦略の選択と根拠の記録
2. 受容の場合: 受容理由・受容者・受容条件・レビュー周期の記録
3. 軽減の場合: 対策内容・実施スケジュール・期待効果の記録
4. 移転の場合: 移転先（保険・外部委託等）・残留リスクの記録
5. 回避の場合: 回避方法・業務影響の記録
6. 承認ワークフロー（リスクスコアに応じた承認レベル）
   - LOW: リスクオーナー承認
   - MEDIUM: 部門長承認
   - HIGH: CISO承認
   - CRITICAL: 経営層承認
7. 承認履歴の記録
8. 承認期限管理・エスカレーション

### RSK-005: 残存リスクモニタリング

| 項目 | 内容 |
|------|------|
| **要件ID** | RSK-005 |
| **要件名** | 残存リスクモニタリング |
| **優先度** | 必須 |
| **カテゴリ** | リスク管理 |

**概要:**
対策実施後の残存リスクの継続的なモニタリングと四半期ごとの再評価。

**詳細要件:**

7. 残存リスクトレンドレポートの自動生成
8. Celery Beatによる定期チェック・通知の自動実行

### RSK-006: 建設業特有リスクテンプレート

| 項目 | 内容 |
|------|------|

## 16. リスクと前提条件

### 16.1 前提条件

1. PostgreSQL 16が利用可能であること
2. Docker/Docker Composeが利用可能であること
3. SSL証明書が取得済みであること（本番環境）
4. SMTPサーバーが利用可能であること（通知機能）
5. ISO27001:2022の管理策データが正確に整備されていること
6. NIST CSF 2.0とISO27001のマッピングデータが整備されていること
7. 各法令の要件チェックリストが関連部門により作成されること

### 16.2 プロジェクトリスク

| リスク | 影響度 | 発生可能性 | 対策 |
|--------|--------|-----------|------|
| 法令要件の解釈相違 | 高 | 中 | 法務部門との定期レビュー |
| ISO27001管理策データの不正確さ | 高 | 低 | 公式文書に基づく精査 |
| 外部システム連携の技術的問題 | 中 | 中 | Phase 3での段階的導入 |
| ユーザー受入れの遅れ | 中 | 中 | 早期のプロトタイプレビュー |
| 性能要件の未達 | 中 | 低 | 早期の性能テスト実施 |
