## 1. アクター定義

### 1.1 アクター一覧

| アクターID | アクター名 | 説明 | 主な責務 |
|-----------|-----------|------|---------|
| ACT-001 | GRC管理者 | システム全体の管理権限を持つ管理者 | システム設定、ユーザー管理、フレームワーク設定、マスタデータ管理 |
| ACT-002 | リスクオーナー | リスクの特定・評価・対応を担当する部門責任者 | リスク登録、評価実施、対応計画策定、モニタリング |
| ACT-003 | コンプライアンス担当 | 法令・規格への準拠管理を担当する専門担当者 | 法令マッピング、準拠状況管理、ギャップ分析、改善計画策定 |
| ACT-004 | 内部監査員 | 内部監査の計画・実施・報告を担当する監査担当者 | 監査計画策定、監査実施、所見記録、是正勧告、フォローアップ |
| ACT-005 | 経営層 | 経営判断に必要な情報を参照する経営者・役員 | ダッシュボード閲覧、レポート確認、承認・決裁 |
| ACT-006 | 一般部門担当 | 各部門のGRC関連業務を実施する一般従業員 | リスク報告、エビデンス登録、是正対応、日常点検 |

## 3. リスク管理ユースケース（UC-RSK）

### 3.1 UC-RSK-001: リスクを登録する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-RSK-001 |
| **ユースケース名** | リスクを登録する |
| **主アクター** | リスクオーナー（ACT-002） |
| **副アクター** | 一般部門担当（ACT-006）、GRC管理者（ACT-001） |
| **概要** | 新たに識別されたリスクをシステムに登録し、初期評価と分類を行う |
| **トリガー** | リスクオーナーがリスク登録画面を開く / 一般部門担当がリスク報告フォームを送信する |
| **頻度** | 日常的（週5〜20件程度） |

**事前条件:**

| # | 条件 |
|---|------|
| PRE-1 | アクターがシステムにログイン済みであること |
| PRE-2 | アクターが「リスク登録」権限を有すること |
| PRE-3 | リスクカテゴリマスタが設定済みであること |

**事後条件:**

| # | 条件 |
|---|------|
| POST-1 | 新規リスクレコードがデータベースに保存されること |
| POST-2 | リスクIDが自動採番されること（RSK-YYYY-NNNN形式） |
| POST-3 | リスクオーナーに通知が送信されること |
| POST-4 | 監査ログに登録操作が記録されること |

**基本フロー:**

| ステップ | アクター | システム |
|---------|---------|---------|
| 1 | リスク管理メニューから「新規登録」を選択 | リスク登録フォームを表示 |
| 2 | リスク名、説明を入力 | 入力内容をリアルタイムバリデーション |
| 3 | リスクカテゴリを選択（情報セキュリティ/労働安全/環境/法令/品質/財務） | 選択に応じたサブカテゴリを表示 |
| 4 | リスクオーナー（担当部門・担当者）を指定 | ユーザー一覧から候補を表示 |
| 5 | 関連する建設プロジェクト・現場を選択 | プロジェクトマスタから候補表示 |
| 6 | 初期リスク評価（発生確率・影響度）を入力 | リスクスコアを自動計算し表示 |
| 7 | 関連するフレームワーク統制項目を紐付け | ISO27001/NIST CSFの統制項目一覧を表示 |
| 8 | 添付ファイル（エビデンス）をアップロード | ファイルをストレージに保存 |
| 9 | 「登録」ボタンを押下 | バリデーション実行後、データベースに保存 |
| 10 | — | リスクIDを自動採番し表示 |
| 11 | — | リスクオーナーにメール/Slack通知送信 |
| 12 | — | 監査ログに操作を記録 |

**代替フロー:**

| ID | 条件 | 処理 |
|----|------|------|
| ALT-1 | バリデーションエラー発生 | エラー箇所をハイライト表示し、修正を促す |
| ALT-2 | 類似リスクが既に登録済み | 類似リスク一覧をダイアログ表示し、重複確認を求める |
| ALT-3 | 一般部門担当が報告する場合 | 簡易フォーム（リスク名・説明・カテゴリのみ）で登録し、リスクオーナーに詳細入力を依頼 |
| ALT-4 | ファイルアップロード失敗 | エラーメッセージ表示、リトライ可能。リスク登録自体は続行可能 |
| ALT-5 | セッションタイムアウト | 入力内容をローカルストレージに一時保存し、再ログイン後に復元 |

**例外フロー:**

| ID | 条件 | 処理 |
|----|------|------|
| EXC-1 | サーバーエラー（500） | エラー画面表示、入力内容をローカルに保存、管理者に自動通知 |
| EXC-2 | ストレージ容量超過 | アップロード不可の旨を表示、管理者に通知 |

### 3.2 UC-RSK-002: リスクを評価する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-RSK-002 |
| **ユースケース名** | リスクを評価する |
| **主アクター** | リスクオーナー（ACT-002） |
| **副アクター** | GRC管理者（ACT-001） |
| **概要** | 登録済みリスクに対して定量的・定性的評価を実施し、リスクレベルを決定する |
| **トリガー** | リスクの定期評価時期到来 / リスク状況の変化検知 |
| **頻度** | 月次（全リスク棚卸）、随時（状況変化時） |

**事前条件:**

| # | 条件 |
|---|------|
| PRE-1 | 対象リスクが「登録済み」または「評価待ち」ステータスであること |
| PRE-2 | リスク評価基準（影響度・発生確率のマトリクス）が設定済みであること |
| PRE-3 | アクターが対象リスクの評価権限を有すること |

**事後条件:**

| # | 条件 |
|---|------|
| POST-1 | リスク評価結果がデータベースに保存されること |
| POST-2 | リスクレベル（極高・高・中・低・極低）が更新されること |
| POST-3 | リスクヒートマップに反映されること |
| POST-4 | 「極高」「高」の場合、経営層に自動通知されること |

**基本フロー:**

| ステップ | アクター | システム |
|---------|---------|---------|
| 1 | リスク一覧から対象リスクを選択 | リスク詳細画面を表示 |
| 2 | 「評価」タブを選択 | 評価フォームを表示（前回評価値をプリセット） |
| 3 | 発生確率を5段階で評価（1:極低〜5:極高） | 評価根拠の入力欄を表示 |
| 4 | 影響度を5段階で評価（財務・運用・法令・安全・信用の各観点） | 各観点の影響度を入力 |
| 5 | 評価根拠・コメントを入力 | — |
| 6 | 現在の統制状況を評価（有効/一部有効/無効） | 残留リスクスコアを自動計算 |
| 7 | 「評価確定」ボタンを押下 | 固有リスク・残留リスクを計算し保存 |
| 8 | — | リスクレベルに基づきヒートマップを更新 |
| 9 | — | 「極高」「高」の場合、経営層・GRC管理者にアラート通知 |

**代替フロー:**

| ID | 条件 | 処理 |
|----|------|------|
| ALT-1 | 評価に必要な情報が不足 | 情報提供リクエストを関連部門に送信可能 |
| ALT-2 | 複数リスクの一括評価 | バッチ評価画面に遷移し、一括で評価入力が可能 |
| ALT-3 | 前回評価からの変更なし | 「変更なし」を選択し、評価日のみ更新 |

### 3.3 UC-RSK-003: リスク対応計画を策定する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-RSK-003 |
| **ユースケース名** | リスク対応計画を策定する |
| **主アクター** | リスクオーナー（ACT-002） |
| **副アクター** | GRC管理者（ACT-001）、一般部門担当（ACT-006） |
| **概要** | 評価済みリスクに対する対応戦略（回避・軽減・移転・受容）と具体的な対応計画を策定する |
| **トリガー** | リスク評価完了後 / 対応計画見直し時期の到来 |
| **頻度** | リスク評価後随時 |

**事前条件:**

| # | 条件 |
|---|------|
| PRE-1 | 対象リスクの評価が完了していること |
| PRE-2 | リスクレベルが「中」以上であること（「低」「極低」は受容前提） |
| PRE-3 | アクターが対応計画策定権限を有すること |

**事後条件:**

| # | 条件 |
|---|------|
| POST-1 | 対応計画がリスクレコードに紐付けて保存されること |
| POST-2 | 対応担当者にタスクが割り当てられること |
| POST-3 | 対応期限がスケジュールに登録されること |
| POST-4 | 対応計画の進捗追跡が可能になること |

**基本フロー:**

| ステップ | アクター | システム |
|---------|---------|---------|
| 1 | リスク詳細画面から「対応計画」タブを選択 | 対応計画フォームを表示 |
| 2 | リスク対応戦略を選択（回避/軽減/移転/受容） | 選択に応じた入力項目を表示 |
| 3 | 具体的な対応施策を入力 | — |
| 4 | 対応担当者を指定 | ユーザー一覧から候補を表示 |
| 5 | 対応期限を設定 | カレンダーから日付選択 |
| 6 | 必要リソース（予算・人員）を入力 | — |
| 7 | 目標残留リスクレベルを設定 | 現在のリスクレベルとの比較を表示 |
| 8 | 「計画登録」ボタンを押下 | データベースに保存、担当者に通知 |
| 9 | — | 対応タスクをタスク管理に自動登録 |
| 10 | — | 期限前リマインダーをスケジュール |

**代替フロー:**

| ID | 条件 | 処理 |
|----|------|------|
| ALT-1 | 受容戦略を選択 | 受容理由と承認者を入力。経営層の承認ワークフローを起動 |
| ALT-2 | 移転戦略を選択 | 移転先（保険会社・外部委託先）の情報入力欄を追加表示 |
| ALT-3 | 既存の統制項目で対応可能 | 統制項目との紐付けのみで計画策定完了 |

### 3.4 UC-RSK-004: リスクをモニタリングする

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-RSK-004 |
| **ユースケース名** | リスクをモニタリングする |
| **主アクター** | リスクオーナー（ACT-002） |
| **副アクター** | GRC管理者（ACT-001）、経営層（ACT-005） |
| **概要** | 登録済みリスクの状況を継続的にモニタリングし、変化を検知・対応する |
| **トリガー** | 定期モニタリングスケジュール / KRI閾値超過アラート |
| **頻度** | 日次（自動チェック）、週次（手動レビュー） |

**事前条件:**

| # | 条件 |
|---|------|
| PRE-1 | モニタリング対象のリスクが登録・評価済みであること |
| PRE-2 | KRI（Key Risk Indicator）が設定済みであること |
| PRE-3 | アラート閾値が設定済みであること |

**事後条件:**

| # | 条件 |
|---|------|
| POST-1 | モニタリング結果が記録されること |
| POST-2 | 閾値超過時にアラートが発報されること |
| POST-3 | リスクステータスが適切に更新されること |

**基本フロー:**

| ステップ | アクター | システム |
|---------|---------|---------|
| 1 | リスクダッシュボードを表示 | リスク概況（件数、レベル分布、トレンド）を表示 |
| 2 | モニタリング対象リスクを選択 | リスク詳細とKRIの推移グラフを表示 |
| 3 | 現在のリスク状況を確認 | 前回評価との差分をハイライト表示 |
