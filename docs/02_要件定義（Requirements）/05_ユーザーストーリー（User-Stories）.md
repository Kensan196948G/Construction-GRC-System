## 1. 概要

本文書では、Construction-GRC-Systemの全ユーザーストーリーを「〜として、〜したい、〜のために」形式で定義する。各ストーリーにはMoSCoW優先度と受入条件を付与し、スプリント計画の入力資料とする。

### 1.1 ストーリーID体系

| プレフィックス | 対象領域 |
|-------------|---------|
| US-RSK | リスク管理 |
| US-CMP | コンプライアンス管理 |
| US-CTL | 統制管理 |
| US-AUD | 監査管理 |
| US-RPT | レポート・ダッシュボード |
| US-SYS | システム管理 |

## 2. MoSCoW優先度定義

| 優先度 | 記号 | 説明 |
|-------|------|------|
| **Must Have** | M | リリースに必須。これがなければシステムが成立しない |
| **Should Have** | S | 重要だが、初回リリースでなくても許容できる |
| **Could Have** | C | あれば望ましいが、なくても業務は回る |
| **Won't Have** | W | 今回スコープ外。将来リリースで検討 |

## 3. リスク管理ストーリー

### US-RSK-001: リスクの新規登録

| 項目 | 内容 |
|------|------|
| **ストーリーID** | US-RSK-001 |
| **優先度** | M（Must Have） |
| **ストーリーポイント** | 8 |
| **スプリント** | Sprint 1-2 |

**ストーリー:**
> **リスクオーナーとして**、新たに識別したリスクをシステムに登録**したい**。**なぜなら**、組織全体のリスクを一元的に可視化し、適切な対応判断を行う基盤を構築するためである。

**受入条件:**

- [ ] リスク名、説明、カテゴリ、発生確率、影響度を入力して登録できる
- [ ] リスクIDが「RSK-YYYY-NNNN」形式で自動採番される
- [ ] リスクカテゴリ（情報セキュリティ/労働安全/環境/法令/品質/財務）から選択できる
- [ ] 登録完了後にリスクオーナーへ通知が送信される
- [ ] 監査ログに登録操作が記録される
- [ ] バリデーションエラー時に適切なエラーメッセージが表示される

---

### US-RSK-002: リスクの評価実施

| 項目 | 内容 |
|------|------|
| **ストーリーID** | US-RSK-002 |
| **優先度** | M（Must Have） |
| **ストーリーポイント** | 8 |
| **スプリント** | Sprint 2-3 |

**ストーリー:**
> **リスクオーナーとして**、登録済みリスクに対して発生確率と影響度を5段階で評価**したい**。**なぜなら**、リスクレベルを定量的に把握し、対応優先度を客観的に判断するためである。

**受入条件:**

- [ ] 発生確率を1〜5の5段階で評価できる
- [ ] 影響度を財務・運用・法令・安全・信用の各観点で5段階評価できる
- [ ] リスクスコア（発生確率 x 影響度）が自動計算される
- [ ] リスクレベル（極高/高/中/低/極低）が自動判定される
- [ ] 「極高」「高」のリスクは経営層に自動通知される
- [ ] 評価履歴が時系列で保存・参照できる
