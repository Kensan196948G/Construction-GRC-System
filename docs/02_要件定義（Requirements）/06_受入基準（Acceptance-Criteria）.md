## 1. 概要

本文書では、Construction-GRC-Systemの全機能に対する受入基準をGiven-When-Then（GWT）形式で定義する。本基準は受入テスト実施時の判定根拠として使用する。

### 1.1 GWT形式の説明

| 要素 | 説明 |
|------|------|
| **Given**（前提条件） | テスト実行前に満たすべき状態・条件 |
| **When**（操作・イベント） | テスト対象の操作またはイベント |
| **Then**（期待結果） | 操作後に期待される結果・状態変化 |

## 2. リスク管理機能（RSK）受入基準

### AC-RSK-001: リスク登録

#### AC-RSK-001-01: 正常なリスク登録

```gherkin
Given リスクオーナーがシステムにログインしている
  And リスクカテゴリマスタに「情報セキュリティ」が登録されている
When リスク登録フォームに以下を入力する:
  | フィールド     | 値                               |
  | リスク名       | 社内ネットワークへの不正アクセス    |
  | 説明           | 外部からの不正アクセスによる情報漏洩リスク |
  | カテゴリ       | 情報セキュリティ                  |
  | 発生確率       | 3                                |
  | 影響度         | 4                                |
  And 「登録」ボタンを押下する
Then リスクが正常に登録される
  And リスクIDが「RSK-2026-0001」形式で自動採番される
  And リスクスコアが「12」と計算される
  And リスクレベルが「高」と判定される
  And リスクオーナーに通知メールが送信される
  And 監査ログに「リスク登録」操作が記録される
```

#### AC-RSK-001-02: 必須項目未入力時のバリデーション

```gherkin
Given リスクオーナーがリスク登録フォームを表示している
When リスク名を空欄のまま「登録」ボタンを押下する
Then バリデーションエラーが表示される
  And エラーメッセージ「リスク名は必須です」が表示される
  And リスクはデータベースに保存されない
```

#### AC-RSK-001-03: 類似リスクの重複警告

```gherkin
Given 「社内ネットワークへの不正アクセス」というリスクが既に登録されている
When リスク名に「ネットワーク不正アクセス」と入力する
Then 類似リスク候補として「社内ネットワークへの不正アクセス」が表示される
  And 類似度スコアが表示される
  And 「新規登録を続行」または「既存リスクを表示」を選択できる
```

#### AC-RSK-001-04: 一般部門担当による簡易リスク報告

```gherkin
Given 一般部門担当がシステムにログインしている
When 簡易リスク報告フォームにリスク名・説明・カテゴリを入力する
  And 「報告」ボタンを押下する
Then リスクが「報告済み」ステータスで登録される
  And リスクオーナーに「詳細入力依頼」通知が送信される
```

### AC-RSK-002: リスク評価

#### AC-RSK-002-01: リスクスコアの自動計算

```gherkin
Given リスク「RSK-2026-0001」が登録済みである
  And リスクオーナーがリスク詳細画面の評価タブを表示している
When 発生確率を「4」、影響度（最大値）を「5」と入力する
  And 「評価確定」ボタンを押下する
Then 固有リスクスコアが「20」と計算される
  And リスクレベルが「極高」と判定される
  And ヒートマップ上の位置が(4,5)セルに更新される
  And 経営層およびGRC管理者にアラート通知が送信される
```

#### AC-RSK-002-02: 残留リスクの計算

```gherkin
Given リスク「RSK-2026-0001」の固有リスクスコアが「20」である
  And 関連する統制の有効性が「有効」と評価されている
When 統制状況を「有効」と選択する
  And 「評価確定」ボタンを押下する
Then 残留リスクスコアが固有リスクスコアより低い値で計算される
  And 残留リスクレベルが表示される
```

#### AC-RSK-002-03: 評価履歴の保存

```gherkin
Given リスク「RSK-2026-0001」の評価が2回実施済みである
When 評価履歴タブを選択する
Then 過去2回分の評価結果が時系列で表示される
  And 各評価の日付、評価者、スコア、レベルが確認できる
  And スコアの推移がグラフで表示される
```

## 3. コンプライアンス管理機能（CMP）受入基準

### AC-CMP-001: コンプライアンス要件登録

#### AC-CMP-001-01: 要件の正常登録

```gherkin
Given コンプライアンス担当がシステムにログインしている
  And ISO27001フレームワークが登録済みである
When コンプライアンス要件登録フォームに以下を入力する:
  | フィールド     | 値                                |
  | 適用法令       | ISO27001:2022                     |
  | 要件名称       | A.8.2 特権的アクセス権            |
  | 要件内容       | 特権的アクセス権の割当て及び利用を制限し管理する |
  | 適用範囲       | 全社                              |
  And 「登録」ボタンを押下する

  And 要件IDが「CMP-2026-0001」形式で自動採番される
```

#### AC-CMP-001-02: CSV一括インポート

```gherkin
Given GRC管理者がCSVインポート画面を表示している
  And ISO27001 Annex A全93管理策を含むCSVファイルを用意している
When CSVファイルをアップロードする
  And 「インポート実行」ボタンを押下する
Then 93件の管理策がコンプライアンス要件として一括登録される
  And インポート結果（成功件数、エラー件数）が表示される
  And エラーがあった場合、エラー詳細がダウンロードできる
```

#### AC-CMP-002-01: 準拠ステータスの設定

#### AC-CMP-002-02: 準拠率の自動計算


#### AC-CMP-002-03: 非準拠項目の自動登録

```gherkin

Then 該当要件が改善計画候補として自動登録される
  And コンプライアンス担当に改善計画策定の通知が送信される
```

### AC-CTL-004: エビデンス管理

#### AC-CTL-004-01: エビデンスアップロード

```gherkin
Given 一般部門担当がシステムにログインしている
  And 統制項目「CTL-ISO-0001」のエビデンスタブを表示している
When 「特権アカウント棚卸結果.xlsx」（5MB）をドラッグ＆ドロップする
  And 説明「2026年Q1 特権アカウント棚卸結果」と入力する
  And 「アップロード確定」ボタンを押下する

  And SHA-256ハッシュ値が計算・記録される

  And 監査ログにアップロード操作が記録される
```

#### AC-CTL-004-02: 不正ファイル形式の拒否

```gherkin
Given 一般部門担当がエビデンスアップロード画面を表示している
When 「malware.exe」ファイルをアップロードしようとする
Then アップロードが拒否される
```

### AC-NFR-004: データ整合性

#### AC-NFR-004-01: 監査ログの不変性

```gherkin
Given 監査ログに100件の操作記録が存在する
When 管理者が監査ログの削除を試みる
Then 削除操作は拒否される
  And エラーメッセージ「監査ログは削除できません」が表示される
  And 削除試行自体が監査ログに記録される
```

#### AC-NFR-004-02: データの論理削除

```gherkin
Given リスク「RSK-2026-0001」が登録されている
When GRC管理者がリスクを「削除」する
Then リスクは論理削除される（is_deleted=true, deleted_at=現在日時）
  And 通常の一覧画面には表示されなくなる
  And 管理者は削除済みデータを復元可能
  And データベースからは物理削除されない
```
