## 1. 目的

本文書は、Construction-GRC-Systemにおける情報セキュリティインシデントの対応計画を定義する。インシデントの検知、分類、対応、復旧、報告の各フェーズにおける手順を確立し、インシデントの影響を最小限に抑えることを目的とする。

---

## 2. 適用範囲

### 2.1 対象インシデント

| カテゴリ | 例 |
|---------|---|
| 不正アクセス | アカウント乗っ取り、権限昇格、不正ログイン |
| データ漏洩 | 個人情報漏洩、GRCデータの外部流出、入札情報の漏洩 |
| マルウェア感染 | ランサムウェア、トロイの木馬、ウイルス |
| サービス妨害（DoS/DDoS） | システム停止、サービス劣化 |
| 内部不正 | データ持出し、権限の不正利用、監査ログの改ざん |
| 設定ミス | アクセス権限の誤設定、セキュリティ設定の不備 |
| 物理的セキュリティ | サーバールームへの不正侵入、機器の盗難 |
| サプライチェーン | 外部委託先からの情報漏洩、ソフトウェアサプライチェーン攻撃 |

### 2.2 インシデントと非インシデントの区別

| 事象 | 分類 | 対応 |
|------|------|------|
| 不正ログイン試行（成功） | インシデント | 本計画に基づき対応 |
| 不正ログイン試行（失敗、少数回） | セキュリティイベント | 監視・記録 |
| マルウェア検知（実行前にブロック） | セキュリティイベント | 記録・分析 |
| 既知の脆弱性の検出 | 脆弱性 | 脆弱性管理プロセスで対応 |
| システム障害（セキュリティ起因でない） | 運用障害 | 障害管理プロセスで対応 |

## 3. インシデント分類

### 3.1 深刻度分類

| 深刻度 | 定義 | 例 | 対応時間 |
|--------|------|---|---------|
| P1（Critical） | システム全体の停止、大規模データ漏洩、法令違反の即時リスク | ランサムウェア感染、全ユーザーデータ漏洩、入札情報の外部流出 | 即時対応（15分以内に初動開始） |
| P2（High） | 主要機能の停止、限定的なデータ漏洩、重大なセキュリティ侵害 | 管理者アカウントの乗っ取り、健康診断データ漏洩、一部機能のDoS | 1時間以内に初動開始 |
| P3（Medium） | 一部機能の影響、軽微なセキュリティ事象 | 一般ユーザーアカウントの不正利用、設定ミスによるアクセス権超過 | 4時間以内に初動開始 |
| P4（Low） | 影響が極めて限定的、情報提供レベル | ポートスキャン検知、不審メール受信、軽微な設定不備 | 翌営業日対応 |

## 4. インシデント対応チーム（CSIRT）

### 4.1 CSIRT構成

| 役割 | 担当者 | 責務 | 連絡先 |
|------|--------|------|--------|
| インシデントコマンダー | CISO | 全体統括、意思決定、外部連絡の承認 | 緊急連絡先参照 |
| テクニカルリード | テックリード | 技術的分析の統括、修復措置の決定 | 緊急連絡先参照 |
| セキュリティアナリスト | セキュリティ担当 | 証拠収集、フォレンジック分析、脅威分析 | 緊急連絡先参照 |
| インフラ担当 | インフラリード | システム制御、封じ込め措置の実施 | 緊急連絡先参照 |
| GRC担当 | GRC管理者 | GRC影響評価、法令準拠性の確認 | 緊急連絡先参照 |
| コミュニケーション担当 | PM/広報 | 内外への情報発信、ステークホルダー対応 | 緊急連絡先参照 |
| 法務担当 | 法務部門 | 法的助言、報告義務の確認 | 緊急連絡先参照 |
| 外部支援 | セキュリティベンダー | 専門的技術支援、フォレンジック調査 | 契約先連絡先参照 |

### 4.2 エスカレーションマトリクス

| 深刻度 | 第1レベル | 第2レベル | 第3レベル |
|--------|----------|----------|----------|
| P1 | セキュリティアナリスト → テクニカルリード（即時） | インシデントコマンダー（15分以内） | 経営層（30分以内） |
| P2 | セキュリティアナリスト → テクニカルリード（30分以内） | インシデントコマンダー（1時間以内） | 経営層（必要に応じて） |
| P3 | セキュリティアナリスト → テクニカルリード（4時間以内） | インシデントコマンダー（必要に応じて） | - |
| P4 | セキュリティアナリスト（翌営業日） | - | - |

### 5.2 Phase 1: 検知・報告

| ステップ | 実施者 | 内容 | SLA |
|---------|--------|------|-----|
| 1-1 | 監視システム/ユーザー | インシデントの検知・報告 | 即時 |
| 1-2 | セキュリティアナリスト | 初期確認（インシデントか否かの判定） | 15分以内 |
| 1-3 | セキュリティアナリスト | インシデントチケット起票 | 確認後即時 |
| 1-4 | セキュリティアナリスト | 深刻度の初期判定 | 確認後即時 |
| 1-5 | セキュリティアナリスト | エスカレーション実施 | 深刻度に応じて |

#### 報告チャネル

| チャネル | 用途 | 対象 |
|---------|------|------|
| Slack #security-incident | インシデント報告・対応コミュニケーション | CSIRT |
| メール: security@grc.example.co.jp | 外部からのインシデント報告 | セキュリティチーム |
| 電話: 緊急連絡網 | P1インシデントの緊急連絡 | CSIRT全メンバー |
| GRCシステム: インシデント報告フォーム | 一般ユーザーからの報告 | 全ユーザー |

### 5.3 Phase 2: トリアージ・分析

| ステップ | 実施者 | 内容 |
|---------|--------|------|
| 2-1 | テクニカルリード | 深刻度の確定、対応チームの編成 |
| 2-2 | セキュリティアナリスト | 影響範囲の特定（影響を受けるシステム、データ、ユーザー） |
| 2-3 | セキュリティアナリスト | 証拠の保全（ログ、メモリダンプ、ディスクイメージ） |
| 2-4 | GRC担当 | GRC影響評価（法令準拠性、管理策への影響） |
| 2-5 | テクニカルリード | 対応方針の決定 |

#### 証拠保全チェックリスト

| # | 証拠 | 保全方法 | 優先度 |
|---|------|---------|--------|
| 1 | システムログ（監査ログ、アクセスログ） | ログのコピー・保全（タイムスタンプ付き） | 最高 |
| 2 | ネットワークトラフィック | パケットキャプチャ | 高 |
| 3 | メモリダンプ | 揮発性メモリの保全 | 高 |
| 4 | ディスクイメージ | フォレンジックイメージの取得 | 中 |
| 5 | 設定ファイル | 現在の設定のバックアップ | 中 |
| 6 | スクリーンショット | 画面の記録 | 低 |

### 5.4 Phase 3: 封じ込め

| 封じ込め措置 | 適用条件 | 実施者 |
|------------|---------|--------|
| アカウントの無効化 | 不正アクセスが確認されたアカウント | システム管理者 |
| ネットワーク隔離 | 感染が疑われるサーバーの隔離 | インフラ担当 |
| サービスの一時停止 | 広範な侵害が確認された場合 | インシデントコマンダー承認 |
| IPアドレスのブロック | 攻撃元IPの特定 | インフラ担当 |
| 権限の一時剥奪 | 権限昇格攻撃の場合 | システム管理者 |
| データアクセスの制限 | データ漏洩が疑われる場合 | システム管理者 |
| VPN/リモートアクセスの遮断 | 外部からの侵入が疑われる場合 | インフラ担当 |

### 5.5 Phase 4: 根絶

| ステップ | 内容 |
|---------|------|
| 4-1 | 根本原因の特定 |
| 4-2 | マルウェアの駆除（感染している場合） |
| 4-3 | 攻撃者のアクセス手段の排除（バックドア除去等） |
| 4-4 | 脆弱性の修正（パッチ適用、設定修正） |
| 4-5 | 認証情報のリセット（侵害されたパスワード、API鍵等） |
| 4-6 | 根絶措置の検証 |

### 5.6 Phase 5: 復旧

| ステップ | 内容 | 確認者 |
|---------|------|--------|
| 5-1 | システムの復旧（バックアップからの復元、再構築） | インフラ担当 |
| 5-2 | サービスの段階的再開 | テクニカルリード |
| 5-3 | 正常性確認（ヘルスチェック、スモークテスト） | QAリード |
| 5-4 | 監視の強化（再発の兆候を監視） | セキュリティアナリスト |
| 5-5 | ユーザーへのサービス再開通知 | コミュニケーション担当 |
| 5-6 | GRC機能の正常性確認 | GRC管理者 |

### 5.7 Phase 6: 事後対応

| ステップ | 内容 | 期限 |
|---------|------|------|
| 6-1 | インシデント報告書の作成 | インシデント収束後5営業日以内 |
| 6-2 | 根本原因分析（RCA）の実施 | インシデント収束後10営業日以内 |
| 6-3 | 再発防止策の策定 | RCA完了後5営業日以内 |
| 6-4 | 改善措置の実施・追跡 | 計画に基づき |
| 6-5 | インシデント対応の振り返り（Post-Incident Review） | インシデント収束後15営業日以内 |
| 6-6 | 対応手順の更新（必要な場合） | 振り返り後10営業日以内 |

## 6. 報告義務

### 6.1 内部報告

| 報告先 | 報告条件 | 報告期限 | 報告内容 |
|--------|---------|---------|---------|
| 経営層 | P1/P2インシデント | P1: 30分以内、P2: 2時間以内 | 概要、影響、対応状況 |
| CISO | 全インシデント | 検知後即時 | 詳細情報 |
| 法務部門 | 法令違反リスクがある場合 | 2時間以内 | 法的リスク評価 |
| 関連部門長 | 所管業務に影響する場合 | 4時間以内 | 影響と対応状況 |

### 6.2 外部報告義務

| 報告先 | 報告条件 | 報告期限 | 根拠法令 |
|--------|---------|---------|---------|
| 個人情報保護委員会 | 個人情報漏洩（1,000件超等） | 速報: 速やか（概ね3〜5日）、確報: 30日以内 | 個人情報保護法第26条 |
| 警察 | サイバー犯罪が疑われる場合 | 速やかに | 刑法等 |
| IPA/JPCERT/CC | 脆弱性情報、攻撃情報の共有 | 速やかに | 情報セキュリティ早期警戒パートナーシップ |
| 国土交通省 | 建設業法関連の重大なセキュリティ事故 | 速やかに | 建設業法 |
| 発注者 | 公共工事関連データへの影響がある場合 | 速やかに | 契約条項 |
| NISC | 重要インフラに該当する場合 | 速やかに | サイバーセキュリティ基本法 |

| 演習種別 | 頻度 | 参加者 | 内容 |
