## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemにおいてセキュリティインシデントが発生した場合の対応計画を定める。6段階の体系的な対応プロセス（検知→分析→封じ込め→根絶→復旧→教訓）により、インシデントの影響を最小化し、迅速なサービス復旧と再発防止を実現する。

### 1.2 適用範囲

| 対象 | 説明 |
|------|------|
| 対象システム | Construction-GRC-System および関連する全てのサブシステム |
| 対象インシデント | 情報セキュリティに関する全てのインシデント |
| 対象者 | 全従業員、外部委託先、CSIRT メンバー |
| 適用環境 | 本番環境、ステージング環境、開発環境 |

### 1.3 関連規格

| 規格 | 関連管理策 | 要件 |
|------|----------|------|
| ISO27001 | A.5.24 | 情報セキュリティインシデント管理の計画及び準備 |
| ISO27001 | A.5.25 | 情報セキュリティ事象の評価及び決定 |
| ISO27001 | A.5.26 | 情報セキュリティインシデントへの対応 |
| ISO27001 | A.5.27 | 情報セキュリティインシデントからの学習 |
| ISO27001 | A.5.28 | 証拠の収集 |
| ISO27001 | A.6.8 | 情報セキュリティ事象の報告 |
| NIST CSF 2.0 | RS.MA | インシデント管理 |
| NIST CSF 2.0 | RS.AN | インシデント分析 |
| NIST CSF 2.0 | RS.CO | インシデント報告・コミュニケーション |
| NIST CSF 2.0 | RS.MI | インシデント軽減 |
| NIST CSF 2.0 | RC.RP | 復旧計画の実行 |
| NIST CSF 2.0 | RC.CO | 復旧コミュニケーション |

### 1.4 インシデントの定義

| 用語 | 定義 |
|------|------|
| セキュリティ事象（Event） | 情報セキュリティに関連する可能性のある事象。インシデントかどうかは未確定 |
| セキュリティインシデント（Incident） | 情報の機密性・完全性・可用性を侵害する、または侵害する可能性が高い事象 |
| 重大インシデント（Major Incident） | 事業運営に重大な影響を与えるセキュリティインシデント |

## 3. インシデント分類

### 3.1 インシデントカテゴリ

| カテゴリ | 説明 | 例 |
|---------|------|---|
| 不正アクセス | 許可されていないシステム/データへのアクセス | 権限昇格、アカウント乗っ取り |
| データ漏洩 | 機密情報の不正な流出 | DB漏洩、ファイル流出、メール誤送信 |
| マルウェア | 悪意あるソフトウェアの感染 | ランサムウェア、バックドア |
| DoS/DDoS | サービス妨害攻撃 | アプリケーションDoS、ネットワークDDoS |
| 脆弱性の悪用 | 既知/未知の脆弱性を利用した攻撃 | SQLi、XSS、RCE |
| 内部不正 | 内部者による意図的な不正行為 | データ持出し、権限の不正利用 |
| 設定ミス | セキュリティ設定の誤りによるインシデント | 公開設定ミス、ファイアウォール設定ミス |
| サプライチェーン | 外部委託先・サプライチェーン経由の攻撃 | ソフトウェアサプライチェーン攻撃 |

### 3.2 重大度分類

| 重大度 | 定義 | 対応体制 | 対応開始目標 |
|--------|------|---------|------------|
| P1: Critical | 事業運営に重大な影響。大規模データ漏洩、システム全停止 | CSIRT全員 + 経営層 | 15分以内 |
| P2: High | 重要な機能の停止。限定的なデータ漏洩、部分的なシステム停止 | CSIRT主要メンバー | 30分以内 |
| P3: Medium | 業務に影響あるが限定的。不審なアクティビティ | インシデントマネージャー + アナリスト | 2時間以内 |
| P4: Low | 業務への直接的な影響なし。ポリシー違反 | セキュリティチーム | 翌営業日 |

### 3.3 建設業固有のインシデント分類

| インシデント種別 | 影響 | 重大度 | 特別な対応 |
|---------------|------|--------|-----------|
| 入札情報の漏洩 | 公正な競争の阻害、行政処分 | P1 | 国土交通省への報告、入札中止の判断 |
| 経審データの改ざん | 不正な評点、許可取消リスク | P1 | 行政機関への報告、データ復旧 |
| 技術者情報の不正利用 | 無資格配置、業法違反 | P2 | 配置変更、行政報告 |
| 安全衛生データの漏洩 | 個人情報漏洩、訴訟リスク | P1 | 個人情報保護委員会への報告、被害者通知 |
| 労災隠しデータの発見 | 法令違反、社会的信用失墜 | P1 | 労基署への報告、内部調査 |

### Phase 1: 検知（Detection）

#### 検知ソース

| 検知ソース | 検知内容 | 監視ツール |
|-----------|---------|-----------|
| 監視システム | リソース異常、パフォーマンス低下 | Prometheus/Grafana |
| ログ分析 | 異常なアクセスパターン、エラーログ | Loki/Grafana |
| IDS/IPS | 不正アクセスの検知・防止 | Suricata/Snort |
| WAF | Webアプリケーション攻撃の検知 | ModSecurity/AWS WAF |
| 脆弱性スキャン | 新規脆弱性の検出 | pip-audit/npm audit/Trivy |
| ユーザー報告 | 不審な動作、フィッシング等 | メール/Slack |
| 外部情報 | 脅威インテリジェンス、JPCERT/CC通知 | 脅威情報フィード |

#### 検知時のアクション

| # | アクション | 責任者 | 時間目標 |
|---|-----------|--------|---------|
| 1 | インシデントの初期記録（日時、検知ソース、概要） | 検知者 | 即時 |
| 2 | インシデントマネージャーへの報告 | 検知者 | 15分以内 |
| 3 | インシデントチケットの作成 | インシデントマネージャー | 30分以内 |
| 4 | 初期重大度の判定 | インシデントマネージャー | 30分以内 |
| 5 | CSIRTメンバーの招集 | インシデントマネージャー | 重大度に応じた時間 |

---

### Phase 2: 分析（Analysis）

#### 分析手順

| # | アクション | 詳細 | 責任者 |
|---|-----------|------|--------|
| 1 | 情報収集 | ログ、アラート、報告内容の収集・整理 | テクニカルアナリスト |
| 2 | 影響範囲の特定 | 影響を受けたシステム、データ、ユーザーの特定 | テクニカルアナリスト |
| 3 | 攻撃ベクトルの分析 | 攻撃手法、侵入経路の特定（MITRE ATT&CK参照） | テクニカルアナリスト |
| 4 | タイムラインの作成 | インシデントの時系列整理 | コミュニケーション担当 |
| 5 | 重大度の再評価 | 分析結果に基づく重大度の見直し | インシデントマネージャー |
| 6 | IoC（Indicator of Compromise）の特定 | 攻撃の痕跡、不正なIP/ドメイン等の特定 | テクニカルアナリスト |
| 7 | GRC影響評価 | 法令準拠性、管理策への影響の確認 | GRC管理者 |

#### 証拠収集・保全（ISO27001 A.5.28）

| # | アクション | ツール |
|---|-----------|--------|
| 1 | メモリダンプの取得 | LiME, Volatility |
| 2 | ディスクイメージの取得 | dd, FTK Imager |
| 3 | ネットワークキャプチャ | tcpdump, Wireshark |
| 4 | ログの保全 | rsync, tar |
| 5 | ハッシュ値の記録 | sha256sum |
| 6 | Chain of Custodyの記録 | フォレンジック台帳 |

---

### Phase 3: 封じ込め（Containment）

#### 短期封じ込め策

| 対策 | 適用条件 | 実行者 |
|------|---------|--------|
| 感染端末のネットワーク隔離 | マルウェア感染 | インフラチーム |
| 不正アカウントの無効化 | アカウント乗っ取り | セキュリティチーム |
| 攻撃元IPのブロック | 外部からの攻撃 | インフラチーム |
| WAFルールの追加 | Webアプリ攻撃 | インフラチーム |
| 脆弱な機能の一時停止 | 脆弱性の悪用 | 開発チーム |
| データベースアクセスの制限 | データ漏洩 | DBA |
| APIレート制限の強化 | DoS攻撃 | インフラチーム |
| 全ユーザーの強制ログアウト | 広範なアカウント侵害 | セキュリティチーム |

#### 長期封じ込め策

| 対策 | 詳細 | 実行者 |
|------|------|--------|
| 代替システムの起動 | バックアップからの復旧環境の構築 | インフラチーム |
| ネットワークセグメントの再設計 | 攻撃経路の遮断 | インフラチーム |
| 認証情報の全リセット | 全パスワード、APIキー、トークンの再発行 | セキュリティチーム |
| 監視の強化 | 追加のログ収集、リアルタイム監視の強化 | セキュリティチーム |

---

### Phase 4: 根絶（Eradication）

#### 根絶手順

| # | アクション | 責任者 |
|---|-----------|--------|
| 1 | 根本原因の特定 | テクニカルアナリスト |
| 2 | マルウェアの除去 | セキュリティチーム |
| 3 | 脆弱性の修正（パッチ適用） | 開発チーム |
| 4 | バックドアの確認・除去 | セキュリティチーム |
| 5 | 設定の修正 | インフラチーム |
| 6 | 認証情報の再生成 | セキュリティチーム |
| 7 | クリーンな環境の構築（必要時） | インフラチーム |

#### 根絶確認チェックリスト

| 確認項目 | 確認方法 | 合格基準 |
|---------|---------|---------|
| マルウェアの完全除去 | アンチマルウェアスキャン | 検出なし |
| 脆弱性の修正 | OWASP ZAP, Trivy等で再スキャン | 該当CVEが検出されない |
| バックドアの不在 | ファイル整合性チェック、ポートスキャン | 不審なファイル/ポートなし |
| 不正アカウントの不在 | ユーザー一覧の監査 | 不正アカウントなし |
| 設定の正常性 | 構成管理ツールによる検証 | ベースラインとの一致 |

## 5. 報告義務

### 5.1 社内報告

| 報告先 | 報告条件 | 報告期限 | 報告者 |
|--------|---------|---------|--------|
| CISO | 全てのセキュリティインシデント | 検知後30分以内 | インシデントマネージャー |
| 経営層（CEO/CTO） | P1, P2インシデント | 検知後1時間以内 | CISO |
| 全従業員 | P1インシデント（影響がある場合） | 封じ込め完了後 | コミュニケーション担当 |

### 5.2 外部報告

| 報告先 | 報告条件 | 報告期限 | 根拠法令 |
|--------|---------|---------|---------|
| 個人情報保護委員会 | 個人情報漏洩（1,000件以上/要配慮個人情報） | 速報: 3〜5日以内 / 確報: 30日以内 | 個人情報保護法第26条 |
| 本人（データ主体） | 個人情報漏洩（権利利益を害するおそれ） | 速やかに | 個人情報保護法第26条 |
| JPCERT/CC | 重大なサイバー攻撃 | 速やかに | 推奨 |
| IPA | 脆弱性情報の発見 | 速やかに | 情報セキュリティ早期警戒パートナーシップ |
| 国土交通省 | 建設業法関連データの重大な漏洩・改ざん | 速やかに | 建設業法 |
| 警察 | サイバー犯罪の被害 | 必要に応じて | 刑法 |
| 顧客・取引先 | 顧客データに影響がある場合 | 影響確認後速やかに | 契約上の義務 |
| NISC | 重要インフラに該当する場合 | 速やかに | サイバーセキュリティ基本法 |

### 5.3 個人情報保護法に基づく報告フロー

```
個人情報漏洩の検知
  │
  ▼
報告義務の対象か？（1000件以上 / 要配慮個人情報 / 不正目的のおそれ）
  │             │
  はい          いいえ → 任意報告の検討（推奨）
  │
  ▼
速報（3〜5日以内）：概要、漏洩件数、対応状況
  │
  ▼
確報（30日以内、不正目的60日以内）：詳細調査結果、再発防止策
  │
  ▼
本人への通知（速やかに）：漏洩情報の内容、対応状況、相談窓口
```
