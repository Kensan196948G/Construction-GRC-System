## 1. 目的

本文書は、Construction-GRC-Systemのアクセス制御設計を定義する。ロールベースアクセス制御（RBAC）のロール定義、権限マトリクス、認証・認可の設計方針を明確化し、情報資産への適切なアクセス制御を実現することを目的とする。

---

## 2. アクセス制御の基本方針

### 2.1 設計原則

| 原則 | 説明 |
|------|------|
| 最小権限の原則（Least Privilege） | ユーザーには業務遂行に必要な最小限の権限のみを付与する |
| 職務分離（Separation of Duties） | 重要な業務を単一の人物に集中させない |
| Need-to-Know | 業務上知る必要のある情報のみアクセスを許可する |
| デフォルト拒否（Default Deny） | 明示的に許可されていないアクセスは全て拒否する |
| 多層防御（Defense in Depth） | 認証、認可、監査の各層で制御を実施する |

### 2.2 アクセス制御モデル

本システムでは、RBAC（Role-Based Access Control）を基本モデルとし、必要に応じてABAC（Attribute-Based Access Control）を組み合わせて使用する。

```
┌──────────────────────────────────────────────────────┐
│                  アクセス制御アーキテクチャ               │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ユーザー ──► 認証（Authentication）                    │
│               │                                      │
│               ├── SAML/OIDC（SSO）                   │
│               ├── ローカル認証                         │
│               └── MFA（多要素認証）                     │
│                       │                              │
│                       ▼                              │
│              認可（Authorization）                     │
│               │                                      │
│               ├── RBAC（ロールベース）                 │
│               │    └── ロール → パーミッション          │
│               │                                      │
│               ├── ABAC（属性ベース：補助）              │
│               │    └── 部署、プロジェクト、機密レベル    │
│               │                                      │
│               └── 行レベルセキュリティ（RLS）           │
│                    └── データ所有者、部門フィルタ        │
│                       │                              │
│                       ▼                              │
│              監査（Audit）                             │
│               └── 全アクセスの監査ログ記録              │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 3.2 ロール別詳細定義

#### ROLE-001: システム管理者（System Administrator）

| 項目 | 内容 |
|------|------|

#### ROLE-004: コンプライアンス担当（Compliance Officer）

### 4.2 データレベルアクセス制御

| データスコープ | 制御方法 | 適用ロール |
|-------------|---------|----------|
| 全社データ | アクセス制限なし | システム管理者、GRC管理者、内部監査員 |
| 部門データ | 行レベルセキュリティ（RLS）で部門フィルタ | リスクオーナー、一般部門担当 |
| プロジェクトデータ | プロジェクトメンバーシップに基づくフィルタ | リスクオーナー、一般部門担当 |
| 個人データ | 本人のみ（+管理者） | 全ロール |

### 4.3 機密レベル別アクセス制御

| 機密レベル | アクセス可能ロール |
|-----------|----------------|
| 極秘 | システム管理者、GRC管理者、経営層 |
| 秘密 | 上記 + 内部監査員、コンプライアンス担当 |
| 社外秘 | 上記 + リスクオーナー、安全衛生管理者 |
| 公開 | 全ロール |

## 5. 認証設計

### 5.1 認証方式

| 認証方式 | 説明 | 適用環境 |
|---------|------|---------|
| SAML 2.0 | 企業IdP（Active Directory等）とのSSO連携 | 本番環境（推奨） |
| OIDC（OpenID Connect） | OAuth 2.0ベースの認証連携 | 本番環境（代替） |
| ローカル認証 | ユーザー名/パスワード + MFA | 開発環境、バックアップ認証 |
| API認証 | JWT（JSON Web Token） + APIキー | API連携 |

### 5.2 パスワードポリシー

| ポリシー項目 | 設定値 |
|------------|--------|
| 最小文字数 | 12文字 |
| 最大文字数 | 128文字 |
| 必須文字種 | 英大文字、英小文字、数字、記号の全てを含む |
| パスワード有効期限 | 90日 |
| パスワード履歴 | 直近12世代の再利用を禁止 |
| ログイン失敗回数 | 5回でアカウントロック（30分） |
| パスワードリセット | メール認証 + 本人確認 |
| 禁止パスワード | 辞書攻撃に使われる一般的なパスワードリスト |

### 5.3 多要素認証（MFA）

| MFA方式 | 優先度 | 対応ロール |
|---------|--------|----------|
| ハードウェアトークン（FIDO2/WebAuthn） | 推奨 | システム管理者 |
| 認証アプリ（TOTP） | 必須対応 | 全ロール |
| SMS認証 | バックアップ | 全ロール |
| メール認証 | バックアップ | 全ロール |

### 5.4 セッション管理

| 設定項目 | 値 | 説明 |
|---------|---|------|
| セッションタイムアウト | 30分（管理者は15分） | アイドル時間超過でセッション終了 |
| 最大セッション時間 | 8時間 | 連続使用の上限 |
| 同時セッション数 | ロール別に設定 | 上記ロール定義参照 |
| セッション固定化対策 | ログイン成功時にセッションID再生成 | セッション固定攻撃防止 |
| セッション情報 | サーバーサイドセッション（Redis） | クライアント側にはセッションIDのみ |
| Cookie設定 | Secure, HttpOnly, SameSite=Strict | Cookie窃取対策 |

## 6. 認可設計

### 6.1 RBAC実装設計

```python
# ロールモデル（概念）
class Role:
    id: str                    # ROLE-001
    name: str                  # "System Administrator"
    description: str           # ロールの説明
    permissions: List[Permission]  # 付与されたパーミッション
    is_active: bool            # 有効/無効
    max_concurrent_sessions: int  # 最大同時セッション数

class Permission:
    id: str                    # PERM-001
    resource: str              # "risk_management"
    action: str                # "create" | "read" | "update" | "delete" | "approve"
    scope: str                 # "all" | "department" | "own"
    conditions: dict           # 追加条件（ABAC用）

class UserRole:
    user_id: str
    role_id: str
    assigned_at: datetime
    assigned_by: str
    valid_from: datetime
    valid_until: datetime       # ロール割当の有効期限
    department_id: str          # 部門スコープ
```

| `/api/v1/admin/settings` | PUT | `system:admin` | システム設定変更 |

## 7. 職務分離（Separation of Duties）

### 7.1 相反する権限の組み合わせ

以下のロールの組み合わせは、同一ユーザーに付与してはならない。

| 組み合わせ | 理由 |
|-----------|------|
| システム管理者 + 内部監査員 | 管理対象の自己監査を防止 |
| GRC管理者 + 内部監査員 | GRC管理の独立した監査を確保 |
| リスクオーナー + 内部監査員 | リスク管理の独立した監査を確保 |
| システム管理者 + GRC管理者 | 技術管理とGRC管理の分離 |

### 7.2 承認ワークフローの分離

| 承認対象 | 申請者 | 承認者 | 最終承認者 |
|---------|--------|--------|----------|
| リスク登録 | リスクオーナー | GRC管理者 | - |
| リスク対応計画 | リスクオーナー | GRC管理者 | 経営層（Critical/Highの場合） |
| コンプライアンス例外 | コンプライアンス担当 | GRC管理者 | 経営層 |
| ユーザー権限変更 | 各部門管理者 | GRC管理者 | システム管理者 |
| システム設定変更 | システム管理者 | CISO | - |

## 8. 権限レビュー

### 8.1 定期レビュースケジュール

| レビュー対象 | 頻度 | 実施者 | 承認者 |
|------------|------|--------|--------|
| 全ユーザーのアクセス権 | 四半期ごと | GRC管理者 | CISO |
| 特権アカウント | 月次 | システム管理者 | CISO |
| 休眠アカウント | 月次 | システム管理者 | GRC管理者 |
| 外部委託先アクセス権 | 月次 | GRC管理者 | CISO |
| ロール定義 | 年次 | GRC管理者 + CISO | 経営層 |

### 8.2 自動検知ルール

| 検知項目 | 条件 | アクション |
|---------|------|----------|
| 休眠アカウント | 30日以上ログインなし | アラート通知、60日で自動無効化 |
| 権限の過剰付与 | 3ロール以上の付与 | GRC管理者にレビュー通知 |
| 相反権限の検出 | 職務分離ルール違反 | 即時アラート、自動ブロック |
| 異常なアクセスパターン | 通常と異なる時間帯/場所からのアクセス | セキュリティチームに通知 |
| 連続ログイン失敗 | 5回連続失敗 | アカウントロック、管理者通知 |

## 9. ユーザーライフサイクル管理

### 9.1 プロビジョニングフロー

```
入社/異動
  │
  ├─► 人事システムからの通知
  │
  ├─► アカウント作成申請
  │    ├── 所属部門
  │    ├── 職位
  │    └── 必要なロール
  │
  ├─► 上長承認
  │
  ├─► GRC管理者承認
  │
  ├─► アカウント作成
  │    ├── ID発行
  │    ├── 初期パスワード設定
  │    ├── ロール割当
  │    └── MFA設定
  │
  ├─► 初期教育
  │    ├── セキュリティポリシー説明
  │    └── システム利用方法教育
  │
  └─► 利用開始
```

### 9.2 デプロビジョニングフロー

```
退職/異動/契約終了
  │
  ├─► 人事システムからの通知
  │
  ├─► 即時アクション
  │    ├── アカウント無効化
  │    ├── セッション強制終了
  │    ├── リモートアクセス権限剥奪
  │    └── 共有パスワード変更（該当する場合）
  │
  ├─► 24時間以内
```
