## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-SystemのRBAC（Role-Based Access Control）設計、6ロールの権限マトリクス、およびJWT + リフレッシュトークン方式の認証設計を定義する。

### 1.2 設計原則

| 原則 | 説明 |
|------|------|
| 最小権限の原則 | 業務遂行に必要な最小限の権限のみ付与 |
| 職務分離 | 相反する職務を同一ユーザに付与しない |
| Need-to-Know | 業務上知る必要のある情報のみアクセス可能 |
| 多層防御 | 認証 + 認可 + 監査の3層で保護 |
| デフォルト拒否 | 明示的に許可されていないアクセスは拒否 |

## 2. RBAC設計

### 2.1 ロール一覧

| ロールID | ロール名 | 説明 | 想定ユーザ数 |
|---------|---------|------|------------|
| R001 | GRC管理者 | システム全体の管理権限、ユーザ管理、設定変更 | 2-3名 |
| R002 | リスクオーナー | リスク管理の作成・編集・承認、担当リスクの管理 | 5-10名 |
| R003 | コンプライアンス担当 | 法令対応管理、是正管理、パトロール管理 | 5-10名 |
| R004 | 内部監査員 | 監査計画・実施・報告、全データの読取アクセス | 3-5名 |
| R005 | 経営層 | ダッシュボード閲覧、レポート承認、経営判断 | 3-5名 |
| R006 | 一般担当 | 限定的な閲覧権限、報告の作成・提出 | 20-50名 |

### 2.2 ロール階層

```
┌─────────────┐
│  GRC管理者   │ ← システム全権限
│   (R001)    │
├─────────────┤
│  内部監査員  │ ← 全データ読取 + 監査管理
│   (R004)    │
├─────────────┤
│  経営層      │ ← ダッシュボード + 承認
│   (R005)    │
├─────────────┤
│ﾘｽｸｵｰﾅｰ(R002)│ ← リスク管理 + 担当範囲の編集
│ｺﾝﾌﾟﾗ (R003) │ ← コンプライアンス管理 + 担当範囲の編集
├─────────────┤
│  一般担当    │ ← 限定閲覧 + 報告提出
│   (R006)    │
└─────────────┘
```

## 4. JWT + リフレッシュトークン設計

### 4.1 認証フロー概要

```
┌──────────┐                    ┌──────────┐
│  Client  │                    │  Server  │
│ (Vue.js) │                    │ (Django) │
└────┬─────┘                    └────┬─────┘
     │                               │
     │  1. POST /api/v1/auth/login   │
     │  {username, password, mfa}    │
     │──────────────────────────────>│
     │                               │ 認証処理
     │  2. Response                  │ MFA検証
     │  {access_token,               │
     │   refresh_token}              │
     │<──────────────────────────────│
     │                               │
     │  3. GET /api/v1/risks/        │
     │  Authorization: Bearer <AT>   │
     │──────────────────────────────>│
     │                               │ トークン検証
     │  4. Response {data}           │ 権限チェック
     │<──────────────────────────────│
     │                               │
     │  (AT期限切れ後)               │
     │  5. POST /api/v1/auth/refresh │
     │  {refresh_token}              │
     │──────────────────────────────>│
     │                               │ RT検証
     │  6. Response                  │ 新AT発行
     │  {access_token}               │
     │<──────────────────────────────│
```

### 4.2 トークン設計

#### 4.2.1 アクセストークン（AT）

| 項目 | 設定値 |
|------|--------|
| 形式 | JWT (JSON Web Token) |
| アルゴリズム | RS256（RSA + SHA-256） |
| 有効期限 | 15分 |
| 格納場所（クライアント） | メモリ（Pinia Store） |
| ペイロード | user_id, role, permissions, exp, iat, jti |

**ペイロード例:**

```json
{
  "user_id": "U001",
  "username": "yamada.taro",
  "role": "risk_owner",
  "permissions": [
    "risk:read", "risk:create", "risk:update",
    "risk_assessment:read", "risk_assessment:create",
    "report:read", "report:export",
    "dashboard:risk"
  ],
  "department": "リスク管理部",
  "exp": 1711461600,
  "iat": 1711460700,
  "jti": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### 4.2.2 リフレッシュトークン（RT）

| 項目 | 設定値 |
|------|--------|
| 形式 | ランダム文字列（UUID v4 + HMAC） |
| 有効期限 | 7日 |
| 格納場所（クライアント） | HttpOnly Secure Cookie |
| 格納場所（サーバー） | Redis（ユーザID + デバイスID紐付け） |
| ローテーション | リフレッシュ時に新RTを発行（旧RTは無効化） |
| 同時セッション | 最大3デバイスまで |

### 4.3 トークン管理

#### 4.3.1 トークン発行

```python
# Django側のトークン発行ロジック（擬似コード）
def issue_tokens(user):
    access_token = jwt.encode({
        'user_id': user.id,
        'role': user.role.name,
        'permissions': user.get_permissions(),
        'exp': datetime.utcnow() + timedelta(minutes=15),
        'iat': datetime.utcnow(),
        'jti': str(uuid.uuid4())
    }, private_key, algorithm='RS256')

    refresh_token = generate_secure_token()
    redis.setex(
        f"refresh_token:{user.id}:{device_id}",
        timedelta(days=7),
        refresh_token_hash
    )

    return access_token, refresh_token
```

#### 4.3.2 トークン失効

| 失効条件 | 処理 |
|---------|------|
| AT期限切れ | クライアントがRTで再取得 |
| RT期限切れ | 再ログイン要求 |
| ログアウト | RT削除、ATをブラックリストに追加 |
| パスワード変更 | 全セッションのRT削除 |
| アカウントロック | 全セッションのRT削除 |
| 管理者による強制失効 | 全セッションのRT削除、ATをブラックリストに追加 |

#### 4.3.3 ATブラックリスト

```python
# 期限切れ前のATを無効化する仕組み
# Redis に jti を格納（AT残存期間のTTL付き）
def blacklist_token(jti, remaining_ttl):
    redis.setex(f"blacklist:{jti}", remaining_ttl, "1")

def is_blacklisted(jti):
    return redis.exists(f"blacklist:{jti}")
```

### 4.4 MFA（多要素認証）

| 項目 | 設定値 |
|------|--------|
| 方式 | TOTP（Time-based One-Time Password） |
| アルゴリズム | HMAC-SHA1 |
| 桁数 | 6桁 |
| 有効期間 | 30秒 |
| 対象ロール | 全ロール（GRC管理者は必須、他は推奨） |
| バックアップコード | 10個（1回使い切り） |

### 4.5 セッション管理

| 項目 | 設定値 |
|------|--------|
| セッションタイムアウト | 30分（無操作時） |
| 最大セッション数 | 1ユーザ3デバイスまで |
| 同時ログイン | 許可（最大3デバイス、超過時は最古セッションを無効化） |
| セッション固定攻撃対策 | ログイン時にセッションIDを再生成 |

## 5. パスワードポリシー

### 5.1 パスワード要件

| 項目 | 要件 |
|------|------|
| 最小文字数 | 12文字 |
| 最大文字数 | 128文字 |
| 複雑性 | 大文字、小文字、数字、記号のうち3種以上 |
| 禁止パターン | ユーザID、メールアドレス、連続文字（aaa, 123） |
| 辞書チェック | 一般的なパスワードリストとの照合 |
| 有効期限 | 90日 |
| 履歴管理 | 過去12回分のパスワードは再利用不可 |
| ロックアウト | 5回連続失敗でアカウントロック（30分） |

### 5.2 パスワード保存

| 項目 | 設定値 |
|------|--------|
| ハッシュアルゴリズム | Argon2id |
| メモリコスト | 64MB |
| 並列度 | 4 |
| イテレーション | 3 |
