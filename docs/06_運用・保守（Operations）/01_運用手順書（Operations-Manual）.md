## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemの日常運用タスク、定期メンテナンス、および障害時の初動対応に関する標準手順を定める。運用担当者が本文書に従って業務を遂行することにより、システムの安定稼働と品質維持を実現する。

### 1.2 対象読者

| 対象者 | 役割 |
|--------|------|
| システム運用担当者 | 日常運用タスクの実施 |
| インフラエンジニア | インフラ保守・障害対応 |
| オンコール担当者 | 障害時の初動対応 |
| 運用管理者 | 運用品質の監督・改善 |

### 1.3 システム構成概要

| コンポーネント | 技術 | 台数 | 備考 |

### 2.2 ログ確認手順

#### 2.2.1 アプリケーションログ

```bash
# Djangoアプリケーションログの確認
tail -f /var/log/grc-system/application.log

# エラーログのフィルタリング（直近24時間）
journalctl -u grc-application --since "24 hours ago" --priority=err

# Gunicornアクセスログ
tail -100 /var/log/grc-system/access.log
```

**確認ポイント:**

| 項目 | 確認内容 | 閾値 |
|------|---------|------|
| エラー件数 | ERRORレベル以上のログ件数 | 10件/日以上で調査 |
| レスポンスタイム | 平均レスポンス時間 | 500ms以上で警告 |
| 5xxエラー | HTTP 5xx系レスポンス | 1件でも発生時は調査 |
| 認証失敗 | ログイン失敗回数 | 同一IPから5回以上でブロック確認 |

#### 2.2.2 インフラログ

```bash
# Nginxエラーログ
tail -100 /var/log/nginx/error.log

# PostgreSQLログ
tail -100 /var/log/postgresql/postgresql-16-main.log

# Redisログ
tail -50 /var/log/redis/redis-server.log

# システムログ
journalctl --since "24 hours ago" --priority=warning
```

#### 2.2.3 セキュリティログ

```bash
# 認証ログ確認
grep "AUTHENTICATION" /var/log/grc-system/security.log | tail -50

# 権限変更ログ
grep "PERMISSION_CHANGE" /var/log/grc-system/audit.log | tail -20

# 不正アクセス検知
grep "SUSPICIOUS" /var/log/grc-system/security.log | tail -20
```

```bash

celery -A grc_system inspect active

# バッチ実行ログの確認
tail -100 /var/log/grc-system/batch.log

# 失敗タスクの確認
celery -A grc_system inspect failed
```

#### 2.3.3 バッチ手動実行手順

```bash

python manage.py run_batch --batch-id=BAT-002 --force

# 全バッチのステータス確認
python manage.py batch_status --all
```

### 2.4 ユーザ管理

#### 2.4.1 ユーザアカウント管理

**ユーザ作成手順:**

1. 管理画面（`/admin/users/`）にGRC管理者でログイン
2. 「新規ユーザ作成」をクリック
3. 必要情報を入力:
   - ユーザID（社員番号ベース）
   - 氏名
   - メールアドレス
   - 所属部署
   - ロール（6種類から選択）
4. 初期パスワードを自動生成
5. 作成完了後、ユーザにメール通知

**ロール一覧:**

| ロール | 権限概要 | 付与承認者 |
|--------|---------|-----------|
| GRC管理者 | 全機能の管理権限 | 情報セキュリティ責任者 |
| リスクオーナー | リスク管理の作成・編集 | GRC管理者 |
| コンプライアンス担当 | 法令対応の管理 | GRC管理者 |
| 内部監査員 | 監査実施・報告 | GRC管理者 |
| 経営層 | ダッシュボード閲覧・承認 | GRC管理者 |
| 一般担当 | 限定的な閲覧・報告 | 部門管理者 |

#### 2.4.2 アカウントロック解除

```bash
# ロックされたアカウントの確認
python manage.py list_locked_users

# アカウントロック解除
python manage.py unlock_user --username=<ユーザID>

# ログインアクティビティ確認
python manage.py user_activity --username=<ユーザID> --days=7
```

#### 2.4.3 定期アカウント棚卸

- **実施頻度:** 四半期ごと（3月、6月、9月、12月）
- **対象:** 全アクティブアカウント
- **確認項目:**
  - 退職者アカウントの無効化確認
  - 異動者の権限変更確認
  - 未使用アカウント（90日以上未ログイン）の確認
  - 特権アカウントの棚卸

## 4. 障害時初動対応

### 4.1 障害検知

障害は以下の経路で検知される:

1. **自動アラート:** Prometheus/Grafanaによる監視アラート（Slack/メール通知）
2. **ユーザ報告:** ヘルプデスク経由のインシデント報告
3. **定期確認:** 日次ログ確認での異常発見

### 4.2 初動対応フロー

```
障害検知
  │
  ├── 影響範囲確認（5分以内）
  │     ├── 利用者への影響有無
  │     ├── データ損失リスク
  │     └── セキュリティリスク
  │
  ├── 障害レベル判定（GRC-OPS-003参照）
  │     ├── P1（緊急）: 即時エスカレーション
  │     ├── P2（重大）: 30分以内にエスカレーション
  │     ├── P3（中度）: 4時間以内に対応
  │     └── P4（軽微）: 次営業日対応
  │
  ├── 関係者への連絡
  │     ├── 運用管理者
  │     ├── 開発チーム（必要時）
  │     └── ステークホルダー（P1/P2時）
  │
  └── 暫定対応実施
        ├── サービス再起動
        ├── フェイルオーバー
        └── 切り戻し
```

### 4.3 緊急連絡先

| 優先度 | 連絡先 | 連絡方法 | 応答目標 |
|--------|--------|---------|---------|
| 1st | オンコール当番 | Slack + 電話 | 15分以内 |
| 2nd | 運用管理者 | Slack + 電話 | 30分以内 |
| 3rd | 開発リード | Slack + 電話 | 30分以内 |
| 4th | プロジェクトマネージャー | 電話 | 1時間以内 |
| 5th | 情報セキュリティ責任者 | 電話 | 1時間以内 |

### 4.4 初動対応チェックリスト

- [ ] 障害発生時刻の記録
- [ ] 障害の症状・影響範囲の確認
- [ ] 障害レベルの判定
- [ ] 関係者への第一報
- [ ] 暫定対応の実施
- [ ] 対応状況の記録開始
- [ ] 復旧見込み時間の連絡
- [ ] 障害対応チケットの作成
