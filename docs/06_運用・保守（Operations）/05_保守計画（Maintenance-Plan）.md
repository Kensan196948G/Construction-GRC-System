## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemの定期保守スケジュール、セキュリティパッチ管理、バージョンアップ計画、およびEOL（End of Life）管理に関する方針と手順を定める。

### 1.2 保守の分類

| 分類 | 説明 | 例 |
|------|------|-----|
| 予防保守 | 障害を未然に防ぐための計画的保守 | パッチ適用、定期点検 |
| 是正保守 | 発見された不具合の修正 | バグ修正、ホットフィックス |
| 適応保守 | 環境変化への対応 | OS更新、法令改正対応 |
| 完全化保守 | 性能改善・機能向上 | パフォーマンスチューニング |

## 3. セキュリティパッチ管理

### 3.1 パッチ分類と適用SLA

| パッチ分類 | CVSS Score | 適用期限 | 承認プロセス |
|-----------|-----------|---------|------------|
| 緊急（Critical） | 9.0-10.0 | 24時間以内 | 緊急承認（CISO口頭承認可） |
| 高（High） | 7.0-8.9 | 72時間以内 | 簡易承認（運用管理者承認） |
| 中（Medium） | 4.0-6.9 | 2週間以内 | 通常承認（変更管理委員会） |
| 低（Low） | 0.1-3.9 | 1ヶ月以内 | 定期メンテナンスで適用 |

### 3.2 パッチ管理フロー

```
1. 脆弱性情報の収集（自動+手動）
   │
2. 影響度評価（CVSSスコア確認）
   │
3. 適用対象の特定
   │
4. テスト環境での検証
   │  ├── 機能テスト
   │  ├── 回帰テスト
   │  └── 性能テスト
   │
5. 承認取得（分類に応じた承認プロセス）
   │
6. 本番適用
   │  ├── バックアップ取得
   │  ├── パッチ適用
   │  └── 動作確認
   │
7. 適用結果の記録
   │
8. 脆弱性管理台帳の更新
```

### 3.3 脆弱性情報ソース

| ソース | 確認頻度 | 担当 |
|--------|---------|------|
| JVN（Japan Vulnerability Notes） | 日次 | セキュリティ担当 |
| NVD（National Vulnerability Database） | 日次 | セキュリティ担当 |
| Django Security Releases | 発表時即時 | 開発リード |
| Ubuntu Security Notices | 日次 | インフラ担当 |
| PostgreSQL Security | 発表時即時 | DBA |
| npm/pip advisory | 日次（自動スキャン） | CI/CD |

### 3.4 パッチ適用手順

#### 3.4.1 OSパッチ適用

```bash
# 1. 適用可能なパッチの確認
sudo apt update
sudo apt list --upgradable

# 2. テスト環境での適用
sudo apt upgrade -y  # テスト環境

# 3. テスト実施
python manage.py test --parallel
npm run test

# 4. 本番適用（メンテナンスウィンドウ内）
sudo apt upgrade -y

# 5. 再起動（必要時）
sudo reboot

# 6. 動作確認
curl -s http://localhost:8000/health/deep/
```

#### 3.4.2 アプリケーション依存パッケージ更新

```bash
# Python依存パッケージ
pip install --upgrade -r requirements.txt
python manage.py test

# Node.js依存パッケージ
npm audit
npm update
npm run test
npm run build
```

## 5. EOL（End of Life）管理

### 5.1 EOL管理方針

- EOL到達6ヶ月前までに後継バージョンへの移行計画を策定する
- EOL到達3ヶ月前までにテスト環境での移行検証を完了する
- EOL到達1ヶ月前までに本番移行を完了する
- EOL後のソフトウェアの使用は原則禁止とする

## 6. 保守体制

### 6.1 保守チーム構成

| 役割 | 担当人数 | 責務 |
|------|---------|------|
| 保守リーダー | 1名 | 保守計画策定・進捗管理 |
| アプリケーション保守 | 2名 | バグ修正・機能改善 |
| インフラ保守 | 2名 | サーバー・ネットワーク保守 |
| DBA | 1名 | データベース保守・最適化 |
| セキュリティ担当 | 1名 | 脆弱性管理・パッチ管理 |

### 6.2 保守品質指標（KPI）

| KPI | 目標値 | 測定頻度 |
|-----|--------|---------|
| システム稼働率 | 99.5%以上 | 月次 |
| セキュリティパッチ適用率（Critical） | 100%（24h以内） | 月次 |
| セキュリティパッチ適用率（High） | 100%（72h以内） | 月次 |
| 障害復旧時間（平均） | RTO 4時間以内 | 四半期 |
| 保守作業起因障害件数 | 0件 | 月次 |
| EOL超過ソフトウェア数 | 0件 | 四半期 |
