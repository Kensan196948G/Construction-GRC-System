## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemの保守計画を定める。定期保守、脆弱性対応、バージョンアップ計画を通じて、システムの安定稼働、セキュリティ維持、継続的な改善を実現する。

### 1.2 保守の種類

| 保守種別 | 説明 | 頻度 |
|---------|------|------|
| 予防保守 | 障害の未然防止を目的とした保守 | 定期実施 |
| 修正保守 | バグ修正、不具合対応 | 随時 |
| 適応保守 | 環境変化への対応（OS更新、法令改正等） | 随時 |
| 完全化保守 | 機能追加、性能改善、ユーザビリティ向上 | 計画的に実施 |

## 3. 脆弱性対応

### 3.1 脆弱性情報の収集

| 情報源 | 確認頻度 | 担当 |
|--------|---------|------|
| GitHub Dependabot アラート | 日次（自動） | 開発チーム |
| pip-audit / npm audit（CI） | 毎PR（自動） | CI |
| Django セキュリティアドバイザリ | 週次 | セキュリティ担当 |
| Vue.js セキュリティアドバイザリ | 週次 | セキュリティ担当 |
| PostgreSQL セキュリティ情報 | 週次 | DBA |
| JPCERT/CC 注意喚起 | 日次 | セキュリティ担当 |
| NVD（National Vulnerability Database） | 週次 | セキュリティ担当 |
| Docker セキュリティ情報 | 週次 | インフラチーム |

### 3.2 脆弱性対応フロー

```
1. 脆弱性情報の受信
   │
2. 影響評価
   ├→ 該当しない → 記録のみ
   └→ 該当する → 3へ
   │
3. 重大度評価（CVSS）
   ├→ CRITICAL (9.0-10.0) → 即時対応（24時間以内）
   ├→ HIGH (7.0-8.9)     → 3営業日以内
   ├→ MEDIUM (4.0-6.9)   → 次スプリント
   └→ LOW (0.1-3.9)      → バックログ
   │
4. 対応実施
   ├→ パッチ適用
   ├→ バージョンアップ
   ├→ ワークアラウンド適用
   └→ WAFルール追加
   │
5. テスト
   ├→ 自動テスト（CI）
   └→ 回帰テスト
   │
6. デプロイ
   │
7. 確認・記録
```

### 3.3 脆弱性対応の優先度

| CVSS スコア | 対応期限 | 承認 | テスト |
|------------|---------|------|--------|
| 9.0-10.0（CRITICAL） | 24時間以内 | 事後承認可 | 最小限の回帰テスト |
| 7.0-8.9（HIGH） | 3営業日以内 | 運用リーダー承認 | 関連機能の回帰テスト |
| 4.0-6.9（MEDIUM） | 次スプリント | 通常のPRレビュー | 通常のCI/CDテスト |
| 0.1-3.9（LOW） | バックログ | 通常のPRレビュー | 通常のCI/CDテスト |

### 3.4 緊急パッチ適用手順

```bash
# 1. 影響範囲の確認
pip audit  # Python
cd frontend && npm audit  # Node.js

# 2. パッチブランチの作成
git checkout -b hotfix/security-patch-CVE-XXXX-YYYY main

# 3. パッケージの更新
pip install --upgrade <affected-package>
# または
cd frontend && npm update <affected-package>

# 4. テスト実行
pytest
cd frontend && npm run test:unit

# 5. PRの作成・緊急レビュー

# 6. マージ・デプロイ

# 7. 適用確認
pip audit  # 脆弱性が解消されたことを確認
```

## 5. 技術的負債管理

### 5.1 技術的負債の分類

| 分類 | 説明 | 例 |
|------|------|-----|
| コード負債 | リファクタリングが必要なコード | 重複コード、複雑なメソッド |
| 依存負債 | 古いバージョンの依存パッケージ | EOLに近いライブラリ |
| テスト負債 | テストカバレッジの不足 | テストのないモジュール |
| ドキュメント負債 | 陳腐化したドキュメント | 更新されていないAPI仕様 |
| インフラ負債 | 最適化されていないインフラ設定 | リソースのオーバープロビジョニング |

### 5.2 技術的負債の管理方法

| 管理項目 | 方法 |
|---------|------|
| 負債の記録 | GitHub Issuesに `tech-debt` ラベルで管理 |
| 優先度付け | 影響度と改善コストで優先度を決定 |
| 計画的な解消 | 各スプリントの20%を技術的負債解消に充当 |
| 定期レビュー | 四半期ごとに技術的負債レビューを実施 |
| メトリクス追跡 | コードカバレッジ、依存パッケージ鮮度、静的解析スコア |

### 5.3 コード品質メトリクス

| メトリクス | 目標値 | 測定ツール | 頻度 |
|-----------|--------|----------|------|
| テストカバレッジ（Backend） | > 80% | pytest-cov | CI毎 |
| テストカバレッジ（Frontend） | > 70% | Vitest | CI毎 |
| 循環的複雑度 | < 10 | Ruff / radon | CI毎 |
| コード重複率 | < 5% | jscpd | 月次 |
| 依存パッケージ脆弱性 | 0（CRITICAL/HIGH） | pip-audit, npm audit | CI毎 |
| リンターエラー | 0 | Ruff, ESLint | CI毎 |
