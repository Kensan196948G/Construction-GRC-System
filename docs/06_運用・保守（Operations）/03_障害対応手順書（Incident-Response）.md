## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemにおけるシステム障害発生時の対応手順を定める。障害レベルの定義、エスカレーションフロー、復旧手順を明確化し、迅速かつ的確な障害対応を可能にする。

### 1.2 対象範囲

- アプリケーション障害
- インフラ障害（サーバー、ネットワーク、ストレージ）
- データベース障害
- 外部サービス連携障害
- セキュリティインシデント（別途GRC-SEC-004参照）

## 2. 障害レベル定義

### 2.1 障害レベル一覧

| レベル | 名称 | 定義 | 対応開始 | 復旧目標 | 報告頻度 |
|--------|------|------|---------|---------|---------|
| P1 | 緊急（Critical） | サービス全面停止、データ損失・破損、セキュリティ侵害 | 即時 | 2時間以内 | 30分ごと |
| P2 | 重大（Major） | 主要機能停止、性能著しい劣化（レスポンス10秒超）、一部ユーザへの影響大 | 30分以内 | 4時間以内 | 1時間ごと |
| P3 | 中度（Moderate） | 一部機能の障害、性能低下（レスポンス3秒超）、回避策あり | 4時間以内 | 1営業日以内 | 4時間ごと |
| P4 | 軽微（Minor） | 軽微なUI不具合、ログエラー（ユーザ影響なし）、改善要望 | 次営業日 | 5営業日以内 | 完了時 |

### 2.2 障害レベル判定基準

#### 2.2.1 P1判定基準（いずれか1つ該当）

- 全ユーザがシステムにアクセスできない
- データベースが停止し、データ損失リスクがある
- セキュリティ侵害が確認または強く疑われる
- 建設現場の安全管理データが閲覧不能（人命に関わる場合）
- 法令報告期限が迫る中、該当機能が使用不能
- バックアップが連続3回以上失敗

#### 2.2.2 P2判定基準（いずれか1つ該当）

- リスク管理モジュールまたはコンプライアンスモジュールが完全停止
- レポート生成機能が停止（経営報告に影響）
- 全体の50%以上のユーザに影響
- データベースレプリケーションが停止
- 認証機能の著しい遅延（新規ログイン不可）

#### 2.2.3 P3判定基準（いずれか1つ該当）

- 特定モジュールの一部機能が利用不可（回避策あり）
- バッチ処理の失敗（手動実行で代替可能）
- 検索機能の著しい遅延
- メール通知の遅延または未送信
- 一部画面の表示崩れ

#### 2.2.4 P4判定基準

- 上記P1-P3に該当しない軽微な不具合
- UIの改善要望
- ログに記録されるがユーザ影響のないエラー
- ドキュメントの誤記

## 3. エスカレーションフロー

### 3.1 エスカレーション体制図

```
┌──────────────────────────────────────────────────────────┐
│                    エスカレーションフロー                    │
│                                                          │
│  Level 1: 運用担当（オンコール）                           │
│     │                                                    │
│     ├── P3/P4 → 自己対応 → 解決報告                       │
│     │                                                    │
│     ├── P2 → Level 2 エスカレーション（30分以内）          │
│     │                                                    │
│     └── P1 → 即時 Level 2+3 同時エスカレーション           │
│                                                          │
│  Level 2: 運用管理者 + 開発リード                          │
│     │                                                    │
│     ├── 対応判断・指示                                     │
│     │                                                    │
│     └── 解決困難 → Level 3 エスカレーション                │
│                                                          │
│  Level 3: プロジェクトマネージャー + CISO                  │
│     │                                                    │
│     ├── 経営報告要否判断                                   │
│     │                                                    │
│     └── 必要時 → Level 4: 経営層報告                      │
│                                                          │
│  Level 4: 経営層                                         │
│     └── 対外発表・法的対応判断                             │
└──────────────────────────────────────────────────────────┘
```

### 3.2 エスカレーション連絡先

| Level | 役割 | 連絡方法 | 応答目標 |
|-------|------|---------|---------|
| L1 | オンコール当番 | Slack: #grc-oncall + 電話 | 15分以内 |
| L2 | 運用管理者 | Slack: #grc-incident + 電話 | 30分以内 |
| L2 | 開発リード | Slack: #grc-incident + 電話 | 30分以内 |
| L3 | プロジェクトマネージャー | 電話 + メール | 1時間以内 |
| L3 | CISO（情報セキュリティ責任者） | 電話 + メール | 1時間以内 |
| L4 | 経営層 | 電話 | 2時間以内 |

### 3.3 エスカレーション判断基準

| 条件 | アクション |
|------|-----------|
| L1対応開始後30分経過で復旧見込みなし | L2へエスカレーション |
| L2対応開始後1時間経過で復旧見込みなし | L3へエスカレーション |
| ユーザ影響が拡大傾向 | 即時上位レベルへエスカレーション |
| セキュリティインシデントの疑い | 即時L3+CISO通知 |
| データ損失が確認された | 即時L3+L4通知 |
| メディア対応が必要な場合 | 即時L4通知 |

## 4. 障害対応手順

### 4.1 共通対応フロー

```
1. 障害検知・受付
   │
2. 初期情報収集（5分以内）
   │  ├── 発生時刻
   │  ├── 影響範囲
   │  ├── 症状の詳細
   │  └── 直近の変更有無
   │
3. 障害レベル判定
   │
4. エスカレーション（レベルに応じて）
   │
5. 障害チケット作成
   │
6. 原因調査・切り分け
   │
7. 暫定対応（復旧優先）
   │
8. 恒久対応（根本原因除去）
   │
9. 復旧確認
   │
10. 障害報告書作成
   │
11. 振り返り・再発防止策
```

### 4.2 アプリケーション障害対応

#### 4.2.1 アプリケーション無応答

```bash
# 1. プロセス状態確認
systemctl status grc-application
ps aux | grep gunicorn

# 2. ログ確認
tail -100 /var/log/grc-system/application.log
journalctl -u grc-application --since "10 minutes ago"

# 3. リソース確認
top -bn1 | head -20
free -m
df -h

# 4. 接続確認
curl -o /dev/null -s -w "%{http_code}" http://localhost:8000/health/

# 5. 暫定復旧（プロセス再起動）
sudo systemctl restart grc-application

# 6. 復旧確認
curl -s http://localhost:8000/health/ | python -m json.tool
```

#### 4.2.2 レスポンス遅延

```bash
# 1. スロークエリ確認
psql -U grc_admin -d grc_db -c "
SELECT pid, now() - pg_stat_activity.query_start AS duration, query
FROM pg_stat_activity
WHERE state = 'active' AND query_start < now() - interval '5 seconds'
ORDER BY duration DESC;"

# 2. Celeryキュー確認
celery -A grc_system inspect active

# 3. Redis確認
redis-cli info memory
redis-cli info clients

# 4. コネクションプール確認
psql -U grc_admin -d grc_db -c "
SELECT count(*) FROM pg_stat_activity WHERE state = 'active';"
```

### 4.3 データベース障害対応

#### 4.3.1 PostgreSQL Primary停止

```bash
# 1. ステータス確認
systemctl status postgresql
pg_isready -h db-primary -p 5432

# 2. ログ確認
tail -100 /var/log/postgresql/postgresql-16-main.log

# 3. 復旧試行
sudo systemctl restart postgresql

# 4. 復旧不可時 → フェイルオーバー
# Replica をPrimaryに昇格
sudo -u postgres pg_ctl promote -D /var/lib/postgresql/16/main

# 5. アプリケーション接続先変更
# Django settings の DATABASE_HOST を Replica に変更
python manage.py dbshell --database=default  # 接続確認
```

#### 4.3.2 レプリケーション遅延・断

```bash
# 1. レプリケーション状態確認
psql -U postgres -c "SELECT * FROM pg_stat_replication;"

# 2. Replica側のステータス確認
psql -h db-replica -U postgres -c "SELECT * FROM pg_stat_wal_receiver;"

# 3. WALの状態確認
psql -U postgres -c "SELECT pg_current_wal_lsn();"

# 4. レプリケーション再構築（必要時）
sudo systemctl stop postgresql  # Replica側
sudo -u postgres pg_basebackup -h db-primary -D /var/lib/postgresql/16/main -P
sudo systemctl start postgresql  # Replica側
```

### 4.4 インフラ障害対応

#### 4.4.1 ディスク容量逼迫

```bash
# 1. 使用状況確認
df -h
du -sh /var/log/* | sort -rh | head -10

# 2. 古いログの圧縮・削除
find /var/log/grc-system/ -name "*.log" -mtime +7 -exec gzip {} \;
find /var/log/grc-system/ -name "*.gz" -mtime +30 -exec rm {} \;

# 3. PostgreSQLの一時ファイル削除
psql -U postgres -c "SELECT pg_size_pretty(pg_database_size('grc_db'));"

# 4. 不要なDockerイメージ削除（Docker環境の場合）
docker system prune -f
```

#### 4.4.2 メモリ不足

```bash
# 1. メモリ状況確認
free -m
ps aux --sort=-%mem | head -20

# 2. OOM Killer確認
dmesg | grep -i "oom\|out of memory"

# 3. キャッシュクリア（最終手段）
sync && echo 3 > /proc/sys/vm/drop_caches

# 4. 原因プロセスの特定と対応
# メモリリークの場合はアプリケーション再起動
sudo systemctl restart grc-application
```
