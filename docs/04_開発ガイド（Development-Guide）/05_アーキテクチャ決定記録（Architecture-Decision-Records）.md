## 1. 概要

本文書は、Construction-GRC-Systemの設計・開発において行われた主要なアーキテクチャ決定を ADR（Architecture Decision Record）形式で記録する。各決定の背景、検討した選択肢、決定理由を明文化し、将来の参照に備える。

### ADR フォーマット

各ADRは以下の構造で記述する。

- **ステータス**: 提案 / 承認 / 非推奨 / 置換
- **コンテキスト**: 決定に至った背景・課題
- **決定**: 選択した方針
- **選択肢**: 検討した候補とその比較
- **根拠**: 決定の理由
- **影響**: 決定によるメリット・デメリット・リスク

## ADR-001: バックエンドフレームワークとして Django を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

建設業GRCシステムのバックエンドフレームワークを選定する必要がある。以下の要件を満たすフレームワークが求められる。

- 複雑なビジネスロジックを表現できるモデル層
- ロールベースアクセス制御（RBAC）の実装容易性
- 監査ログ・変更履歴の記録
- 帳票・レポート生成機能
- 管理画面の迅速な構築
- 国内開発者の採用・学習容易性

### 決定

**Django 5.x + Django REST Framework** を採用する。

### 検討した選択肢

| 選択肢 | 概要 |
|--------|------|
| **Django + DRF** | Python製フルスタックWebフレームワーク + REST API拡張 |
| FastAPI | Python製高速APIフレームワーク |
| Ruby on Rails | Ruby製フルスタックWebフレームワーク |
| Spring Boot | Java製エンタープライズフレームワーク |
| NestJS | TypeScript製サーバーサイドフレームワーク |

### 比較評価

| 評価項目 | Django + DRF | FastAPI | Rails | Spring Boot | NestJS |
|---------|-------------|---------|-------|-------------|--------|
| 管理画面 | 標準搭載 | なし | あり | なし | なし |
| ORM | Django ORM（成熟） | SQLAlchemy（別途） | ActiveRecord | JPA/Hibernate | TypeORM |
| 認証・認可 | 標準搭載 | 別途構築 | Devise | Spring Security | Passport |
| 監査ログ | django-auditlog等 | 別途構築 | paper_trail | Spring Data Envers | 別途構築 |
| 日本語情報 | 豊富 | 中程度 | 豊富 | 豊富 | 少ない |
| 開発速度 | 高速 | 高速 | 高速 | 中程度 | 中程度 |
| 型安全性 | mypy対応 | 標準対応 | 限定的 | 標準対応 | 標準対応 |
| エコシステム | 非常に充実 | 成長中 | 充実 | 非常に充実 | 成長中 |

### 根拠

1. **管理画面の標準搭載**: Django Adminにより初期の管理機能を迅速に構築できる。GRCシステムではマスタデータ管理が多く、管理画面の充実は大きなメリットとなる。
2. **認証・認可の充実**: Django標準の認証基盤に加え、`django-guardian`（オブジェクトレベル権限）、`djangorestframework-simplejwt`（JWT認証）等のライブラリが成熟している。
3. **監査証跡の実装容易性**: `django-auditlog`、`django-simple-history` 等により、モデルの変更履歴を容易に記録できる。GRCシステムにおいて監査証跡は必須機能である。
4. **国内開発者の採用**: Python/Djangoは国内での知名度が高く、開発者の採用が比較的容易である。
5. **バッテリー同梱思想**: フォーム処理、バリデーション、セッション管理、キャッシュ、メール送信等が標準で搭載されており、追加ライブラリの選定・評価コストを削減できる。

### 影響

**メリット**:
- 高速な開発立ち上げ
- 管理画面による迅速なプロトタイピング
- 成熟したエコシステムによる安定性
- セキュリティパッチの迅速な提供

**デメリット**:
- 非同期処理の標準サポートが限定的（Celeryで補完）
- モノリシックアーキテクチャへの誘導（意識的な設計が必要）

**リスク**:
- Django ORMのパフォーマンス限界（複雑なクエリ時）→ 必要に応じて生SQLまたはクエリ最適化で対応

## ADR-002: フロントエンドフレームワークとして Vue.js 3 + Vuetify 3 を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

バックエンドAPIと連携するSPA（Single Page Application）のフロントエンドフレームワークを選定する必要がある。以下の要件を満たすことが求められる。

- データテーブル、フォーム、チャート等のリッチなUI
- 業務アプリケーションに適したUIコンポーネントライブラリ
- TypeScriptサポート
- 学習曲線の緩やかさ

### 決定

**Vue.js 3（Composition API）+ Vuetify 3** を採用する。

### 検討した選択肢

| 選択肢 | UIライブラリ | 概要 |
|--------|------------|------|
| **Vue.js 3 + Vuetify 3** | Vuetify 3 | Material Design準拠のコンポーネント |
| React + MUI | Material UI | React向けMaterial Designライブラリ |
| React + Ant Design | Ant Design | エンタープライズ向けUIライブラリ |
| Angular + Angular Material | Angular Material | Google公式のMaterial実装 |
| Vue.js 3 + Quasar | Quasar | クロスプラットフォーム対応 |

### 比較評価

| 評価項目 | Vue.js + Vuetify | React + MUI | React + AntD | Angular + Material |
|---------|-----------------|-------------|--------------|-------------------|
| 学習曲線 | 緩やか | 中程度 | 中程度 | 急 |
| データテーブル | 高機能（標準） | 別途（AG Grid等） | 高機能 | 基本的 |
| フォーム処理 | VeeValidate連携 | React Hook Form | Ant Form | Reactive Forms |
| TypeScript | 良好 | 優秀 | 良好 | 標準 |
| バンドルサイズ | 中程度 | 中程度 | 大きい | 大きい |
| 日本語情報 | 豊富 | 非常に豊富 | 中程度 | 中程度 |
| エンタープライズ適性 | 高い | 高い | 非常に高い | 非常に高い |

### 根拠

1. **Vuetify 3の業務アプリ適性**: データテーブル（ソート、フィルタ、ページネーション）、フォームバリデーション、ダイアログ等の業務アプリケーションに必要なコンポーネントが網羅されている。
2. **学習曲線の緩やかさ**: Vue.jsはテンプレート構文がHTML/CSSに近く、フロントエンド経験の浅いメンバーでも短期間で習得可能である。
3. **Composition APIによる再利用性**: Vue 3のComposition APIにより、ロジックの再利用性が大幅に向上する。GRCシステムでは類似のCRUD操作が多く、composableパターンが有効である。
4. **SFC（単一ファイルコンポーネント）**: テンプレート、スクリプト、スタイルが1ファイルにまとまるため、コンポーネントの把握が容易である。
5. **エコシステムの統一性**: Vue Router（ルーティング）、Pinia（状態管理）、VeeValidate（バリデーション）等、Vue公式または準公式のライブラリで統一的な開発体験を提供できる。

### 影響

**メリット**:
- Vuetifyによる統一的なUI/UX
- 短い学習期間によるチームの迅速な立ち上げ
- Composition APIによるコードの再利用性向上

**デメリット**:
- ReactやAngularと比較して大規模アプリの事例がやや少ない
- Vuetify 3のカスタマイズに制限がある場合がある

## ADR-003: データベースとして PostgreSQL を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

GRCシステムのデータ永続化に使用するRDBMSを選定する必要がある。取り扱うデータの特性として以下がある。

- リスク評価データ（数値計算、集計）
- コンプライアンス規則・条文（全文検索）
- 監査証跡（大量の変更履歴データ）
- 組織階層データ（再帰的構造）
- JSONフォーマットの柔軟なメタデータ

### 決定

**PostgreSQL 16** を採用する。

### 検討した選択肢

| 選択肢 | 概要 |
|--------|------|
| **PostgreSQL 16** | 高機能オープンソースRDBMS |
| MySQL 8 | 広く普及したオープンソースRDBMS |
| MariaDB 11 | MySQLのフォーク |
| Oracle Database | 商用エンタープライズRDBMS |
| SQL Server | Microsoft製商用RDBMS |

### 比較評価

| 評価項目 | PostgreSQL | MySQL | Oracle | SQL Server |
|---------|-----------|-------|--------|------------|
| JSON サポート | JSONB（高機能） | JSON（基本的） | JSON | JSON |
| 全文検索 | 標準搭載 | 別途（Elasticsearch等） | Oracle Text | Full-Text Search |
| 再帰クエリ（CTE） | 完全サポート | サポート | サポート | サポート |
| パーティショニング | 宣言的パーティション | サポート | 高度 | サポート |
| ライセンス | PostgreSQL License（自由） | GPL | 商用ライセンス | 商用ライセンス |
| Django サポート | 第一級サポート | サポート | サポート（要ドライバ） | サポート（要ドライバ） |
| 拡張性 | 非常に高い | 限定的 | 高い | 中程度 |

### 根拠

1. **JSONB型のサポート**: リスク評価の詳細データやカスタムフィールドなど、スキーマが柔軟に変わるデータをJSONBで効率的に格納・検索できる。
2. **全文検索機能**: コンプライアンス規則の検索に、外部検索エンジンを導入せず PostgreSQL 標準の全文検索で対応可能。日本語検索は `pg_trgm` + `pg_bigm` で対応する。
3. **Django との親和性**: DjangoはPostgreSQLを第一級データベースとしてサポートしており、`ArrayField`、`JSONField`、`HStoreField` 等のPostgreSQL固有フィールドが利用可能。
4. **再帰クエリ**: 組織階層、リスクカテゴリ階層の表現にCTE（Common Table Expressions）による再帰クエリが有効。
5. **ライセンスコスト**: オープンソースであり、ライセンスコストが不要。商用サポートが必要な場合はEDB等の選択肢がある。
6. **パーティショニング**: 監査ログ等の大量データは日付ベースのパーティショニングにより効率的に管理可能。

### 影響

**メリット**:
- 高度なデータ型と検索機能による柔軟なデータモデリング
- Djangoの全機能を活用可能
- コスト効率の高い運用

**デメリット**:
- MySQLと比較してチューニングの知識が必要
- レプリケーション構成がやや複雑

## ADR-004: 非同期タスク処理に Celery + Redis を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

以下の非同期処理要件がある。

- レポート生成（PDF/Excel、数秒～数分）
- 通知メール配信（大量の宛先への一斉送信）
- コンプライアンス期限チェック（定期バッチ）
- リスクスコアの一括再計算
- データインポート/エクスポート
- 外部システム連携

### 決定

**Celery 5.x**（タスクキュー）+ **Redis 7.x**（メッセージブローカー兼キャッシュ）を採用する。

### 検討した選択肢

| 選択肢 | 概要 |
|--------|------|
| **Celery + Redis** | Python製分散タスクキュー + インメモリKVS |
| Celery + RabbitMQ | Celery + AMQPメッセージブローカー |
| Django Q | Django統合の軽量タスクキュー |
| Dramatiq | Python製タスクキュー（Celery代替） |
| Huey | 軽量Python製タスクキュー |

### 根拠

1. **Celeryのエコシステム**: Django連携（`django-celery-beat`、`django-celery-results`）が成熟しており、定期タスク管理をDjango Admin経由で行える。
2. **Redisの多目的利用**: メッセージブローカーとしてだけでなく、キャッシュバックエンド（Django Cache Framework）、セッションストア、リアルタイム通知（Pub/Sub）としても利用でき、インフラ構成をシンプルに保てる。
3. **スケーラビリティ**: ワーカー数の動的調整、タスクの優先度制御、リトライ機構が標準で搭載されている。
4. **監視**: Flower（Webベース監視ツール）により、タスクの実行状況をリアルタイムで監視できる。

### 影響

**メリット**:
- ユーザー体験の向上（重い処理をバックグラウンド化）
- スケーラブルな非同期処理基盤
- Redis の多目的利用によるインフラ簡素化

**デメリット**:
- Celery の設定・デバッグが複雑になることがある
- Redis のデータ永続化には注意が必要（AOF/RDB設定）

## ADR-005: コンテナ化基盤として Docker を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

開発環境の統一、CI/CDパイプラインの構築、本番デプロイの効率化のためにコンテナ技術を導入する必要がある。

### 決定

**Docker** + **Docker Compose** を開発・CI環境で使用する。本番環境ではDocker コンテナとしてデプロイする。

### 根拠

1. **環境の再現性**: 開発者のローカル環境、CI環境、本番環境で同一のコンテナイメージを使用することで「自分の環境では動く」問題を排除する。
2. **開発環境構築の効率化**: `docker compose up` コマンド一発で、全ての依存サービス（PostgreSQL, Redis, Celery等）を含む開発環境を構築できる。
3. **CI/CDとの統合**: GitHub ActionsからDockerイメージをビルド・テスト・デプロイするパイプラインを構築しやすい。
4. **マルチサービス構成**: バックエンド、フロントエンド、Celeryワーカー、データベース等のサービスを個別にスケール可能。

### 影響

**メリット**:
- 環境構築時間の大幅短縮（新メンバーのオンボーディング）
- CI/CDパイプラインとの自然な統合
- 本番環境との差分最小化

**デメリット**:
- Docker自体の学習コスト
- ローカル開発時のリソース消費（メモリ・ディスク）
- ホットリロードの設定に注意が必要

## ADR-006: CI/CD に GitHub Actions を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

コードの品質維持と自動デプロイのためのCI/CDツールを選定する必要がある。ソースコード管理にGitHubを使用しているため、GitHubとの統合が重要な要件となる。

### 決定

**GitHub Actions** を採用する。

### 検討した選択肢

| 選択肢 | 概要 |
|--------|------|
| **GitHub Actions** | GitHub統合のCI/CDサービス |
| CircleCI | クラウドCI/CDサービス |
| Jenkins | セルフホスト型CI/CDサーバー |
| GitLab CI/CD | GitLab統合のCI/CDサービス |
| AWS CodePipeline | AWS統合のCI/CDサービス |

### 根拠

1. **GitHubとのネイティブ統合**: PR、Issue、Releaseとの連携がシームレスで、追加のWebhook設定やOAuth連携が不要。
2. **YAMLベースの設定**: ワークフローの定義がリポジトリ内で管理でき、コードレビューの対象となる。
3. **マーケットプレイスの充実**: 公開アクション（actions/checkout, docker/build-push-action等）により、一般的なタスクの実装が容易。
4. **コスト効率**: パブリックリポジトリは無料、プライベートリポジトリも月2,000分の無料枠がある。
5. **セルフホストランナー**: 必要に応じて自社サーバーでジョブを実行でき、セキュリティ要件に対応可能。

### 影響

**メリット**:
- 追加のインフラ構築が不要
- GitHubのPR/Issueとの統合されたワークフロー
- 豊富なマーケットプレイスアクション

**デメリット**:
- GitHub依存（ベンダーロックイン）
- 実行時間の制限（6時間/ジョブ）
- セルフホストランナーの運用コスト（使用する場合）

## ADR-007: API 設計スタイルとして REST を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

フロントエンドとバックエンド間のAPI通信スタイルを決定する必要がある。

### 決定

**REST API**（Django REST Framework）を採用する。将来的に必要に応じてGraphQL（Strawberry）の部分導入を検討する。

### 検討した選択肢

| 選択肢 | 概要 |
|--------|------|
| **REST API** | リソース指向のHTTP API |
| GraphQL | クエリ言語ベースのAPI |
| gRPC | Protocol BuffersベースのRPC |
| REST + GraphQL ハイブリッド | CRUDはREST、集約クエリはGraphQL |

### 根拠

1. **Django REST Frameworkの成熟度**: シリアライザー、ビューセット、ルーター、フィルタリング、ページネーション等が高度に抽象化されており、CRUDアプリケーションの迅速な開発が可能。
2. **チームの習熟度**: RESTは広く普及しており、チームメンバー全員が基本的な知識を有している。
3. **ツールの充実**: Swagger/OpenAPI（drf-spectacular）による自動API仕様書生成、Postman等のテストツールとの親和性が高い。
4. **キャッシュの容易さ**: HTTPキャッシュヘッダーを活用した標準的なキャッシュ戦略を適用可能。

### 影響

**メリット**:
- 迅速な開発（DRFの自動化機能）
- 標準的なHTTPキャッシュの活用
- OpenAPIによるAPI仕様書の自動生成

**デメリット**:
- オーバーフェッチ/アンダーフェッチの問題（ダッシュボード等の複合データ取得時）→ カスタムエンドポイントで対応

## ADR-008: 認証方式として JWT を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

SPA（フロントエンド）とAPI（バックエンド）間の認証方式を決定する必要がある。

### 決定

**JWT（JSON Web Token）** 認証を採用する。ライブラリとして `djangorestframework-simplejwt` を使用し、アクセストークンはメモリ管理、リフレッシュトークンは httpOnly Cookie で管理する。

### 根拠

1. **ステートレスな認証**: サーバー側でセッション状態を保持する必要がなく、水平スケールが容易。
2. **SPAとの親和性**: フロントエンドから Authorization ヘッダーでトークンを送信するパターンが標準的。
3. **トークンローテーション**: アクセストークン（短寿命）とリフレッシュトークン（長寿命）の組み合わせにより、セキュリティと利便性を両立する。
4. **httpOnly Cookie**: リフレッシュトークンを httpOnly Cookie で管理し、XSS攻撃によるトークン窃取を防止する。

### トークン設計

| トークン種別 | 有効期限 | 保存場所 | 用途 |
|------------|---------|---------|------|
| アクセストークン | 60分 | メモリ（JavaScript変数） | API認証 |
| リフレッシュトークン | 24時間 | httpOnly Cookie | アクセストークンの更新 |

### 影響

**メリット**:
- ステートレスによるスケーラビリティ
- マイクロサービス化への対応容易性

**デメリット**:
- トークンの即時無効化が困難（ブラックリスト方式で対応）
- トークンサイズがセッションIDより大きい

## ADR-009: 状態管理に Pinia を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

Vue.js 3アプリケーションのグローバル状態管理ライブラリを選定する必要がある。

### 決定

**Pinia** を採用する。

### 根拠

1. **Vue 3の公式推奨**: PiniaはVue.js公式の状態管理ライブラリであり、長期的なサポートが期待できる。
2. **Composition APIとの親和性**: Composition API スタイルでストアを定義でき、コードベースの一貫性を保てる。
3. **TypeScriptサポート**: 型推論が優秀で、追加の型定義が最小限で済む。
4. **DevToolsサポート**: Vue DevToolsとの統合により、状態の変化をリアルタイムで追跡できる。
5. **シンプルなAPI**: Vuex（前世代）と比較してボイラープレートが少なく、mutations の概念が不要。

## ADR-010: テストフレームワークの選定

| 項目 | 内容 |
|------|------|
| ステータス | 承認 |
| 決定日 | 2026-03-26 |
| 決定者 | アーキテクチャチーム |

### コンテキスト

バックエンド・フロントエンド双方のテストフレームワークを選定する必要がある。

### 決定

| レイヤー | テストフレームワーク | 用途 |
|---------|-------------------|------|
| バックエンド（単体・結合） | **pytest** + pytest-django | Pythonテスト |
| バックエンド（ファクトリ） | **factory_boy** | テストデータ生成 |
| フロントエンド（単体） | **Vitest** | コンポーネント・ユニットテスト |
| フロントエンド（コンポーネント） | **Vue Test Utils** | Vueコンポーネントテスト |
| E2E | **Playwright** | ブラウザE2Eテスト |
| API | **pytest** + DRF APIClient | APIテスト |
| パフォーマンス | **Locust** / **k6** | 負荷テスト |

### 根拠

1. **pytest**: Pythonのデファクトスタンダードであり、フィクスチャ機構、パラメータ化テスト、プラグインエコシステムが充実している。
2. **Vitest**: Vite ベースの高速テストランナーで、Vue.js 3 + TypeScript との統合が優秀。Jest互換APIにより移行も容易。
3. **Playwright**: クロスブラウザ対応のE2Eテストフレームワーク。Cypress と比較して複数タブ/ウィンドウのテスト、ネットワークインターセプトが強力。
4. **factory_boy**: Django モデルのテストデータ生成にファクトリパターンを適用し、テストの可読性と保守性を向上。
