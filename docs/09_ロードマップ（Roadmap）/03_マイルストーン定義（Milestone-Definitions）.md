## 4. ゲートレビュー基準

### 4.1 ゲートレビュープロセス

```
1. レビュー準備（レビュー1週間前）
   │  ├── 成果物の完成・提出
   │  ├── テスト結果の集計
   │  └── レビュー資料の作成
   │
2. ゲートレビュー会議
   │  ├── 成果物レビュー
   │  ├── テスト結果確認
   │  ├── リスク・課題確認
   │  ├── 品質指標の確認
   │  └── Go/No-Go判定
   │
3. レビュー後アクション
   │  ├── Go → 次フェーズ開始
   │  ├── Conditional Go → 条件付き承認、課題対応と並行
   │  └── No-Go → 追加作業・再レビュー日設定
```

### 4.2 レビュー参加者

| 役割 | 参加者 | 責務 |
|------|--------|------|
| レビュー主催 | プロジェクトマネージャー | 会議進行、最終判断 |
| 技術レビュア | テクニカルリード | 技術的品質の評価 |
| 品質レビュア | QAリード | テスト結果・品質指標の報告 |
| セキュリティレビュア | CISO / セキュリティ担当 | セキュリティ要件の充足確認 |
| ビジネスレビュア | プロダクトオーナー | ビジネス要件の充足確認 |
| 運用レビュア | 運用管理者 | 運用性・保守性の評価 |

### 4.3 品質ゲート基準

| カテゴリ | 基準 | 必須/推奨 |
|---------|------|----------|
| テストカバレッジ | ユニットテスト 80%以上 | 必須 |
| テスト合格率 | 全テスト合格 | 必須 |
| セキュリティ | SAST: Critical/High 0件 | 必須 |
| セキュリティ | DAST: Critical/High 0件 | 必須 |
| セキュリティ | 依存パッケージの既知脆弱性 0件（Critical） | 必須 |
| パフォーマンス | 平均レスポンスタイム 500ms以下 | 必須 |
| パフォーマンス | P99レスポンスタイム 3秒以下 | 必須 |
| コード品質 | Lintエラー 0件 | 必須 |
| コード品質 | コードレビュー承認率 100% | 必須 |
| ドキュメント | API仕様書が最新 | 必須 |

## 5. Go/No-Go判定基準

### 5.1 判定マトリクス

| 判定 | 条件 | アクション |
|------|------|-----------|
| **Go** | 全必須基準を満たす、重大な残課題なし | 次フェーズに進行 |
| **Conditional Go** | 必須基準を概ね満たす（軽微な未達あり）、1週間以内に解決可能 | 条件付きで次フェーズ進行、並行して課題解決 |
| **No-Go** | 必須基準の重大な未達がある、2週間以上の追加作業が必要 | 次フェーズ延期、追加作業計画策定、再レビュー日設定 |

### 5.2 No-Go時のエスカレーション

| 条件 | アクション | 承認者 |
|------|-----------|--------|
| 1回目のNo-Go | 1-2週間の追加開発期間設定 | プロジェクトマネージャー |
| 2回連続No-Go | プロジェクト計画の見直し | プロジェクトスポンサー |
| スケジュール全体に影響 | フェーズ統合・スコープ調整 | ステアリングコミッティ |
