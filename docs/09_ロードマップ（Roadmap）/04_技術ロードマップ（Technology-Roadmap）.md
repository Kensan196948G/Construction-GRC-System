## 3. アーキテクチャ進化計画

### 3.1 Phase1-2：モノリシック（モジュラーモノリス）

```
┌──────────────────────────────────────┐
│            Next.js Frontend          │
│  (React Components + SSR)            │
└──────────────┬───────────────────────┘
               │ HTTP/REST
┌──────────────┴───────────────────────┐
│            NestJS Backend            │
│  ┌──────┐ ┌──────┐ ┌──────┐        │
│  │Auth  │ │Master│ │Risk  │        │
│  │Module│ │Module│ │Module│        │
│  └──────┘ └──────┘ └──────┘        │
│  ┌──────┐ ┌──────┐                 │
│  │Notify│ │Report│                 │
│  │Module│ │Module│                 │
│  └──────┘ └──────┘                 │
└──────────────┬───────────────────────┘
               │
       ┌───────┴───────┐
       │  PostgreSQL   │
       │  + Redis      │
       └───────────────┘
```

**選定理由：**
- 初期フェーズでは開発効率を重視
- モジュール境界を明確にし、将来の分離に備える
- チーム間の調整コストを最小化

### 3.2 Phase3-5：モジュラーモノリス（分離準備）

```
┌──────────────────────────────────────┐
│            Next.js Frontend          │
└──────────────┬───────────────────────┘
               │ HTTP/REST
┌──────────────┴───────────────────────┐
│          API Gateway (NestJS)        │
│  ┌──────────────────────────────┐    │
│  │      Shared Kernel           │    │
│  │  (Auth, Logging, Events)     │    │
│  └──────────────────────────────┘    │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌────┐ │
│  │Risk  │ │Compl.│ │Audit │ │Ext.│ │
│  │Domain│ │Domain│ │Domain│ │Int.│ │
│  └──────┘ └──────┘ └──────┘ └────┘ │
└──────────────┬───────────────────────┘
               │
       ┌───────┼───────┐
       │       │       │
    ┌──┴──┐ ┌──┴──┐ ┌──┴──┐
    │ PG  │ │Redis│ │ S3  │
    └─────┘ └─────┘ └─────┘
```

**進化ポイント：**
- ドメイン境界の明確化（Bounded Context）
- イベント駆動のモジュール間通信の導入
- 各ドメインモジュールの独立デプロイ準備

### 3.3 2028年度以降：マイクロサービス化（段階的）

```
┌──────────────────────────────────────────────────┐
│                Next.js Frontend (BFF)             │
└─────────┬─────────┬─────────┬─────────┬──────────┘
          │         │         │         │
   ┌──────┴──┐ ┌───┴────┐ ┌─┴──────┐ ┌┴────────┐
   │  Risk   │ │Compliance│ │ Audit  │ │  AI/ML  │
   │ Service │ │ Service  │ │Service │ │ Service │
   └────┬────┘ └────┬────┘ └───┬────┘ └────┬────┘
        │           │          │            │
   ┌────┴────┐ ┌────┴────┐ ┌──┴──────┐ ┌──┴──────┐
   │  PG DB  │ │  PG DB  │ │  PG DB  │ │ Vector  │
   └─────────┘ └─────────┘ └─────────┘ │   DB    │
                                         └─────────┘
         ┌─────────────────────────┐
         │   Event Bus (SNS/SQS)   │
         └─────────────────────────┘
```

**マイクロサービス化の判断基準：**
- 各ドメインの開発チームが独立してデプロイしたい場合
- 特定ドメインのスケーリング要件が他と大きく異なる場合
- AI/MLサービスのように異なる技術スタックが必要な場合

## 7. 技術負債管理

### 7.1 技術負債の予防策

| 施策 | 内容 | 実施頻度 |
|------|------|----------|
| コーディング規約の厳守 | ESLint/Prettier自動適用 | 毎コミット |
| SonarQube品質ゲート | 技術的負債比率5%以下 | 毎ビルド |
| 依存関係更新 | Dependabotによる自動PR | 週次 |
| リファクタリングスプリント | 技術負債解消専用スプリント | 四半期に1回 |
| アーキテクチャレビュー | 設計の健全性レビュー | 月次 |

### 7.2 技術負債の分類と対応方針

| 種別 | 対応方針 | 許容レベル |
|------|---------|-----------|
| 意図的・短期的 | 次スプリントで解消 | Issue登録必須 |
| 意図的・長期的 | 四半期のリファクタリングスプリントで解消 | 技術ロードマップに記載 |
| 非意図的 | 発見次第Issue登録、優先度に応じて対応 | SonarQubeで自動検知 |
| 環境起因 | 定期的なアップグレード計画で対応 | ライフサイクル管理 |
