## 1. 品質管理方針

### 1.1 品質ビジョン

「セキュリティ・信頼性・使いやすさの三位一体を実現し、建設業のGRC業務を安心・安全に支えるシステムを提供する」

### 1.2 品質方針

- 品質はプロジェクト全体を通じて継続的に管理する
- 品質コストは予防コストに重点を置き、障害コストを最小化する
- 自動テストを最大限に活用し、反復可能な品質保証を実現する
- セキュリティは品質の最重要要素として位置づける
- ユーザビリティを重視し、利用者満足度を品質指標に含める

## 4. レビュー体制

### 4.1 レビュー種別

| レビュー種別 | 対象 | 参加者 | 頻度 | 基準 |
|------------|------|--------|------|------|
| コードレビュー | 全PR | 開発チーム（最低2名） | 毎PR | コーディング規約準拠、テスト有無 |
| アーキテクチャレビュー | 設計変更 | TL、シニアエンジニア | 随時 | 設計原則準拠、性能・保守性 |
| セキュリティレビュー | セキュリティ関連変更 | セキュリティ担当、TL | 随時 | OWASP Top 10、セキュリティ要件 |
| デザインレビュー | UI/UX変更 | FEチーム、業務チーム | スプリント毎 | デザインガイドライン、アクセシビリティ |
| スプリントレビュー | スプリント成果物 | 全チーム | 隔週 | 受入基準、DoD |
| フェーズゲートレビュー | フェーズ成果物一式 | SC、全チーム | フェーズ毎 | ゲート通過基準 |

### 4.2 コードレビューチェックリスト

| # | チェック項目 | 必須/推奨 |
|---|------------|----------|
| 1 | コーディング規約に準拠しているか | 必須 |
| 2 | 適切な単体テストが追加されているか | 必須 |
| 3 | エラーハンドリングが適切か | 必須 |
| 4 | セキュリティ上の問題がないか（SQLインジェクション、XSS等） | 必須 |
| 5 | パフォーマンスへの悪影響がないか | 必須 |
| 6 | ログ出力が適切か（個人情報の非出力確認含む） | 必須 |
| 7 | APIの入力バリデーションが適切か | 必須 |
| 8 | 変数・関数名が適切か | 推奨 |
| 9 | 不要なコメント・デッドコードが残っていないか | 推奨 |
| 10 | ドキュメント（JSDoc等）が更新されているか | 推奨 |

### 4.3 Definition of Done（DoD）

| # | 完了条件 | 対象レベル |
|---|---------|-----------|
| 1 | コードがコーディング規約に準拠している | 全タスク |
| 2 | 単体テストが作成され、全てパスしている | 全タスク |
| 3 | コードレビューが完了し、指摘事項が解消されている | 全タスク |
| 4 | CIパイプラインが正常に完了している | 全タスク |
| 5 | SonarQubeのQuality Gateをパスしている | 全タスク |
| 6 | セキュリティスキャンでCritical/High脆弱性がない | 全タスク |
| 7 | APIドキュメント（OpenAPI）が更新されている | API変更時 |
| 8 | ステージング環境でのテストが完了している | スプリント成果物 |
| 9 | ユーザーストーリーの受入基準を全て満たしている | スプリント成果物 |
| 10 | パフォーマンステストで基準値を満たしている | フェーズ成果物 |

## 5. テスト戦略

### 5.1 テストレベル

| テストレベル | 目的 | 実施者 | 自動化率目標 | ツール |
|------------|------|--------|-------------|--------|
| 単体テスト | 個々の関数・コンポーネントの動作確認 | 開発者 | 100% | Jest, React Testing Library |
| 結合テスト | モジュール間の連携確認 | QAチーム | 80% | Jest, Supertest |
| APIテスト | APIエンドポイントの動作確認 | QAチーム | 90% | Postman, Newman |
| E2Eテスト | ユーザーシナリオの端到端確認 | QAチーム | 70% | Playwright |
| パフォーマンステスト | 性能・負荷の確認 | QAチーム | 90% | k6 |
| セキュリティテスト | 脆弱性の検出 | セキュリティ担当 | 80% | OWASP ZAP, Snyk |
| ユーザビリティテスト | 使いやすさの確認 | 業務チーム | 手動 | - |
| ユーザー受入テスト（UAT） | 業務要件の充足確認 | 業務チーム | 手動 | - |

### 5.2 テストピラミッド

```
        /\
       /  \        ← E2Eテスト（少数の重要シナリオ）
      /    \
     /------\
    /        \     ← 結合テスト / APIテスト
   /          \
  /------------\
 /              \  ← 単体テスト（最も多い）
/________________\
```

| テストレベル | 比率 | 実行時間目安 |
|------------|------|------------|
| 単体テスト | 70% | 数秒〜数分 |
| 結合/APIテスト | 20% | 数分〜十数分 |
| E2Eテスト | 10% | 十数分〜30分 |

### 5.3 テスト環境

| 環境名 | 用途 | 更新頻度 | データ |
|--------|------|----------|--------|
| ローカル開発環境 | 開発者ごとの開発・テスト | 随時 | モックデータ |
| CI環境 | 自動テスト実行 | 毎コミット | テスト用フィクスチャ |
| ステージング環境 | 結合テスト、UAT | 毎スプリント | 本番相当マスクデータ |
| パフォーマンステスト環境 | 負荷テスト | フェーズ毎 | 大量テストデータ |
| 本番環境 | 本番運用 | リリース時 | 本番データ |

### 5.4 障害重大度の定義

| 重大度 | 定義 | 対応期限 | 例 |
|--------|------|----------|-----|
| Critical | システム停止、データ損失、セキュリティ侵害 | 4時間以内 | 認証機能の停止、データ漏洩 |
| Major | 主要機能の利用不可（回避策なし） | 24時間以内 | リスク台帳の登録不可 |
| Minor | 一部機能の不具合（回避策あり） | 次スプリント | 検索結果の並び順不正 |
| Trivial | 外観上の問題、軽微な表示ずれ | バックログ管理 | 文字の配置ずれ |

## 7. 品質改善プロセス

### 7.1 継続的改善サイクル

```
Plan → Do → Check → Act

Plan:  品質目標・メトリクス設定
Do:    開発・テスト実施
Check: メトリクス収集・分析、品質レビュー
Act:   改善策の策定・実施、プロセス改善
```

### 7.2 振り返り（レトロスペクティブ）

| 項目 | 内容 |
|------|------|
| 頻度 | 毎スプリント末 |
| 参加者 | 開発チーム全員 |
| 所要時間 | 1時間 |
| フォーマット | KPT（Keep / Problem / Try） |
| 記録 | Confluence（品質改善台帳） |
| フォローアップ | 次スプリントで改善アクションを実施 |
