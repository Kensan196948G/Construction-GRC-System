## 1. 全体計画概要

### 1.1 プロジェクト方針

本プロジェクトは、みらい建設工業の統合GRCシステムを5つのフェーズに分割し、段階的に構築・導入する。開発手法としてアジャイル（スクラム）を基本とし、各フェーズの中で2週間スプリントを繰り返す。ただし、フェーズ間のゲートレビューではウォーターフォール的な品質確認を行うハイブリッドアプローチを採用する。

### 1.2 開発方針

| 項目 | 方針 |
|------|------|
| 開発手法 | アジャイル（スクラム）＋フェーズゲート |
| スプリント期間 | 2週間 |
| リリース戦略 | フェーズごとの段階リリース |
| 技術スタック | TypeScript / React / Node.js / PostgreSQL |
| インフラ | AWS（東京リージョン） |
| CI/CD | GitHub Actions |
| コード管理 | GitHub（プライベートリポジトリ） |
| テスト戦略 | 自動テスト80%以上カバレッジ |
| セキュリティ | DevSecOps、SAST/DAST組み込み |

## 2. フェーズ構成

### 2.1 フェーズ全体像

```
Phase1: 基盤構築        [2026/04 - 2026/09] 6ヶ月
Phase2: リスク管理      [2026/07 - 2026/12] 6ヶ月（Phase1と一部並行）
Phase3: コンプライアンス [2026/10 - 2027/03] 6ヶ月（Phase2と一部並行）
Phase4: 監査・レポート   [2027/01 - 2027/06] 6ヶ月（Phase3と一部並行）
Phase5: 外部連携        [2027/04 - 2027/09] 6ヶ月（Phase4と一部並行）
```

### 2.2 Phase1：基盤構築（2026年4月〜2026年9月）

**目的：** システム全体の共通基盤を構築し、以降のフェーズの土台を確立する

| 主要成果物 | 内容 |
|-----------|------|
| 認証・認可基盤 | RBAC（ロールベースアクセス制御）、SSO、MFA |
| 共通マスタ管理 | 組織・拠点・プロジェクト・ユーザーマスタ |
| 共通UI基盤 | デザインシステム、レスポンシブ対応 |
| API基盤 | RESTful API、API Gateway、認証トークン管理 |
| データベース基盤 | スキーマ設計、マイグレーション基盤、監査ログ |
| インフラ基盤 | AWS環境構築、CI/CDパイプライン、監視基盤 |
| 通知基盤 | メール通知、Slack連携、アラート管理 |
| ダッシュボード基盤 | 共通ウィジェット、カスタマイズ可能レイアウト |

### 2.3 Phase2：リスク管理（2026年7月〜2026年12月）

**目的：** ISO31000準拠のリスク管理プロセスをシステム化する

| 主要成果物 | 内容 |
|-----------|------|
| リスク台帳管理 | リスクの登録・分類・評価・一覧管理 |
| リスクアセスメント | 定性・定量評価、ヒートマップ表示 |
| リスク対応計画 | 対応策の策定・進捗管理・有効性評価 |
| リスクモニタリング | KRI（重要リスク指標）設定・自動監視 |
| リスクレポート | 経営報告用ダッシュボード、定期レポート自動生成 |
| 建設特有リスク | 工事リスク、安全リスク、環境リスクのテンプレート |

### 2.4 Phase3：コンプライアンス管理（2026年10月〜2027年3月）

**目的：** 建設業関連法令およびISO27001/NIST CSF 2.0の遵守状況を管理する

| 主要成果物 | 内容 |
|-----------|------|
| 法令管理台帳 | 適用法令の一覧管理、条文データベース |
| 遵守状況評価 | 法令ごとの遵守状況セルフチェック機能 |
| ISO27001管理 | 全93管理策のSoA管理、実装状況追跡 |
| NIST CSF 2.0対応 | 6機能の成熟度評価、ギャップ分析 |
| 建設業法対応 | 技術者配置管理、施工体制台帳、下請管理 |
| 品確法対応 | 品質記録管理、検査記録追跡 |
| 労安法対応 | 安全衛生計画、事故報告、安全パトロール記録 |
| 法改正追跡 | 法改正情報の自動収集・影響分析 |

### 2.5 Phase4：監査・レポート（2027年1月〜2027年6月）

**目的：** 内部監査プロセスの効率化と経営レポーティングの自動化を実現する

| 主要成果物 | 内容 |
|-----------|------|
| 監査計画管理 | 年間監査計画の策定・リスクベースの対象選定 |
| 監査実施支援 | チェックリスト生成、証跡収集、監査調書作成 |
| 指摘事項管理 | 指摘の登録・分類・是正措置管理 |
| フォローアップ | 是正状況追跡、期限管理、エスカレーション |
| 経営ダッシュボード | リアルタイムGRCスコアカード |
| 定期報告書 | 月次・四半期・年次レポート自動生成 |
| 規制当局向けレポート | 各種届出書類の自動生成支援 |

## 5. リスク管理方針

### 5.1 リスク管理プロセス

1. **リスク識別** - 週次で新規リスクを識別、リスク登録簿に記録
2. **リスク分析** - 影響度×発生確率でリスクレベルを評価
3. **リスク対応** - 回避・軽減・移転・受容のいずれかの対応策を策定
4. **リスク監視** - 月次でリスクレビュー、ステータス更新

### 5.2 リスクレベル定義

| レベル | スコア | 対応方針 |
|--------|--------|----------|
| 極高 | 20-25 | 即座に対応策実施、SC報告必須 |
| 高 | 12-19 | 1週間以内に対応策策定 |
| 中 | 6-11 | 月次レビューで対応策検討 |
| 低 | 1-5 | 受容可能、モニタリング継続 |

### 5.3 主要リスクと対応策（上位5件）

| # | リスク | 影響度 | 発生確率 | リスクレベル | 対応策 |
|---|--------|--------|----------|-------------|--------|
| 1 | 要件変更の頻発によるスコープクリープ | 5 | 4 | 極高(20) | 変更管理プロセスの厳格化、スコープベースライン管理 |
| 2 | キーメンバーの離任 | 4 | 3 | 高(12) | ナレッジ共有体制、バックアップ要員の確保 |
| 3 | 外部ベンダーの品質問題 | 4 | 3 | 高(12) | ベンダー選定基準の厳格化、品質チェックポイント |
| 4 | セキュリティ要件の追加 | 3 | 4 | 高(12) | セキュリティバイデザイン、早期要件確定 |
| 5 | 法改正によるシステム修正 | 3 | 3 | 中(9) | 法改正モニタリング体制、柔軟なアーキテクチャ |

## 7. 変更管理

### 7.1 変更管理プロセス

1. **変更要求** - 変更依頼書の提出（誰でも可能）
2. **影響分析** - PM/TLによるスコープ・スケジュール・コストへの影響評価
3. **承認判定** - 影響度に応じた承認ルート
   - 小規模（1人日未満）：PM承認
   - 中規模（1-5人日）：PM＋TL承認
   - 大規模（5人日以上）：SC承認
4. **実施・検証** - 承認後の変更実施と結果確認
5. **記録** - 変更管理台帳への記録

### 7.2 スコープベースライン

スコープベースラインは本計画書の承認をもって確定とし、以降の変更は上記プロセスに従う。
