## 1. コミュニケーション方針

### 1.1 基本方針

- 透明性のある情報共有を推進し、プロジェクトの進捗・課題・リスクをタイムリーに伝達する
- 階層に応じた適切な粒度・頻度でコミュニケーションを行う
- 双方向のコミュニケーションを重視し、フィードバックの仕組みを確保する
- セキュリティに配慮した情報管理を行う（機密区分に応じたアクセス制御）

### 1.2 コミュニケーションツール

| ツール | 用途 | 対象者 | 備考 |
|--------|------|--------|------|
| Slack | 日常コミュニケーション、アラート通知 | 全チーム | チャンネル運用ルール別途定義 |
| Microsoft Teams | ステアリングコミッティ、外部ステークホルダー | SC、経営層 | 録画・議事録保存 |
| Jira | タスク管理、進捗管理 | 開発チーム、PMO | スプリントボード運用 |
| Confluence | ドキュメント管理、ナレッジ共有 | 全チーム | 文書テンプレート整備 |
| GitHub | コードレビュー、技術的ディスカッション | 開発チーム | PR/Issueベースの議論 |
| メール | 公式通知、外部連絡 | 全ステークホルダー | 重要決定事項の記録 |
| Miro | ワークショップ、ブレインストーミング | 全チーム | オンラインホワイトボード |

### 2.2 会議運営ルール

| ルール | 内容 |
|--------|------|
| アジェンダ | 会議の24時間前までにアジェンダを共有する |
| 議事録 | 会議終了後24時間以内にConfluenceに議事録を掲載する |
| タイムキーピング | 設定時間を厳守する。延長が必要な場合は参加者の合意を得る |
| 決定事項 | 会議で決定した事項は議事録に明記し、担当者と期限を設定する |
| 欠席時 | 代理出席者を指名するか、事前に意見を文書で提出する |
| 録画 | SC会議は録画し、欠席者が後日確認できるようにする |

## 3. 報告体制

### 3.1 定期報告

| # | 報告書名 | 頻度 | 作成者 | 提出先 | 内容 | 提出期限 |
|---|---------|------|--------|--------|------|----------|
| 1 | デイリーレポート | 日次 | 各チームメンバー | Slack | 本日の作業、翌日の予定、障害 | 当日17:00 |
| 2 | 週次進捗報告書 | 週次 | PMO | PM、SC | WBS進捗、課題、リスク、翌週計画 | 金曜17:00 |
| 3 | 月次プロジェクト報告書 | 月次 | PM | SC | EVM指標、品質メトリクス、予算消化 | 月末5営業日後 |
| 4 | スプリント完了報告 | 隔週 | スクラムマスター | PM | ベロシティ、完了ストーリー、バーンダウン | スプリント最終日 |
| 5 | フェーズ完了報告書 | フェーズ毎 | PM | SC | 成果物一覧、品質評価、次フェーズ計画 | フェーズ完了1週間後 |
| 6 | 品質報告書 | 月次 | QAリーダー | PM | テスト進捗、障害状況、メトリクス | 月末3営業日後 |

### 3.2 報告書テンプレート（週次進捗報告）

```
■ 週次進捗報告書
報告期間：YYYY/MM/DD - YYYY/MM/DD
作成者：PMO

1. 全体サマリー
   - 全体ステータス：🟢緑 / 🟡黄 / 🔴赤
   - スケジュール：SPI = X.XX
   - コスト：CPI = X.XX
   - 品質：障害件数 Critical:X, Major:X

2. 今週の実績
   - 完了タスク一覧
   - マイルストーン達成状況

3. 翌週の計画
   - 予定タスク一覧
   - 注意事項

4. 課題・リスク
   - 新規課題
   - 継続課題（ステータス更新）
   - 新規リスク

5. 承認事項・決定依頼
   - 決定が必要な事項
```

## 4. エスカレーションルール

### 4.1 エスカレーションレベル

| レベル | 条件 | エスカレーション先 | 対応期限 |
|--------|------|-------------------|----------|
| Level 1 | 日常的な課題・判断 | チームリーダー → PM | 当日中 |
| Level 2 | スケジュール1週間以上の遅延見込み | PM → SC（メール報告） | 翌営業日 |
| Level 3 | 予算超過5%以上の見込み | PM → SC（臨時会議） | 48時間以内 |
| Level 4 | プロジェクト目標の達成困難 | PM → SC → 社長 | 24時間以内 |
| Level 5 | セキュリティインシデント | PM → CISO → 社長 | 即時 |

### 4.2 エスカレーションフロー

```
担当者 → チームリーダー → PM → SC → 経営層
              ↓                ↓        ↓
         Level 1           Level 2-3   Level 4-5
         (当日中)          (48h以内)   (24h以内)
```

### 4.3 緊急連絡体制

| 緊急度 | 連絡手段 | 連絡先 | 対応時間 |
|--------|---------|--------|----------|
| 通常（営業時間内） | Slack → メール | チームリーダー | 9:00-18:00 |
| 緊急（営業時間内） | Slack → 電話 | PM | 9:00-18:00 |
| 緊急（営業時間外） | 電話 → SMS | PM → CIO | 24/365 |
| セキュリティインシデント | 電話 | CSIRT窓口 → PM → CIO | 24/365 |

## 7. 情報管理ルール

### 7.1 機密区分

| 区分 | 定義 | 取り扱い | 例 |
|------|------|----------|-----|
| 極秘 | 漏洩した場合に重大な損害が生じる情報 | 配布先限定、暗号化必須 | セキュリティ脆弱性情報 |
| 社外秘 | 社外への開示が不適切な情報 | 社内のみ共有、外部共有時は承認必要 | プロジェクト計画書、設計書 |
| 関係者限定 | プロジェクト関係者のみに共有する情報 | プロジェクトメンバーのみ | 議事録、進捗報告書 |
| 一般 | 特に制限のない情報 | 自由に共有可能 | 公開ドキュメント |

### 7.2 ドキュメント管理

| ルール | 内容 |
|--------|------|
| 保管場所 | Confluence（ドキュメント）、GitHub（ソースコード） |
| 命名規則 | [文書番号]_[文書名]_v[バージョン] |
| バージョン管理 | メジャー変更は整数、マイナー変更は小数で管理 |
| 承認フロー | 作成者 → レビュー者 → 承認者 |
| アクセス権限 | 機密区分に応じたアクセス権設定 |
| 保存期間 | プロジェクト完了後5年間 |
