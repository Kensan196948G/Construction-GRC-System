## 1. 概要

本文書は、建設業統合GRCシステム構築プロジェクトの遂行上のリスク（プロジェクトリスク）を管理するための計画を定義する。本文書で扱うリスクは、システムが管理対象とする業務リスク・コンプライアンスリスクではなく、プロジェクトを成功裏に完了させるために管理すべきリスクである。

---

## 2. リスク管理方針

### 2.1 リスク管理の目的

- プロジェクトの目標（スコープ・スケジュール・コスト・品質）に対する脅威を最小化する
- プロジェクトの機会を最大化する
- ステークホルダーに対するリスクの透明性を確保する
- リスクに対するプロアクティブな対応を推進する

### 2.2 リスク管理プロセス

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  リスク   │ → │  リスク   │ → │  リスク   │ → │  リスク   │ → │  リスク   │
│  識別    │   │  分析    │   │  評価    │   │  対応    │   │  監視    │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
      ↑                                                           │
      └───────────────── 継続的サイクル ──────────────────────────┘
```

## 3. リスク識別

### 3.1 リスク識別手法

| 手法 | 実施時期 | 参加者 |
|------|----------|--------|
| ブレインストーミング | プロジェクト開始時、各フェーズ開始時 | 全チーム |
| チェックリスト法 | 各フェーズ開始時 | PM、TL |

| ステークホルダーインタビュー | 要件定義フェーズ | PM、業務チーム |

### 3.2 リスクカテゴリ

| カテゴリ | サブカテゴリ | 説明 |
|----------|------------|------|
| **技術リスク** | アーキテクチャ | 技術選定の誤り、スケーラビリティ問題 |
| | パフォーマンス | 性能要件未達、レスポンス遅延 |
| | セキュリティ | 脆弱性の発見、セキュリティインシデント |
| | 技術的負債 | コード品質の劣化、保守性の低下 |
| **マネジメントリスク** | スコープ | 要件変更、スコープクリープ |
| | スケジュール | 遅延、クリティカルパスの問題 |
| | コスト | 予算超過、見積精度の問題 |
| | リソース | 人員不足、スキル不足 |
| **外部リスク** | ベンダー | 外部ベンダーの品質・納期問題 |
| | 法規制 | 法改正、規制強化 |
| | 市場 | 技術トレンドの変化、競合製品の出現 |
| **組織リスク** | ステークホルダー | 経営層のコミットメント低下 |
| | 組織変更 | 人事異動、組織再編 |
| | 文化 | 変革への抵抗、利用部門の非協力 |

## 4. リスク分析・評価

### 4.1 影響度の定義

| レベル | スコア | スケジュール影響 | コスト影響 | 品質影響 | スコープ影響 |
|--------|--------|----------------|-----------|---------|------------|
| 極小 | 1 | 1週間未満の遅延 | 予算の1%未満 | 軽微な品質低下 | 些細なスコープ変更 |
| 小 | 2 | 1-2週間の遅延 | 予算の1-5% | 一部機能の品質低下 | 小規模スコープ変更 |
| 中 | 3 | 2-4週間の遅延 | 予算の5-10% | 主要機能の品質低下 | 中規模スコープ変更 |
| 大 | 4 | 1-2ヶ月の遅延 | 予算の10-20% | 重大な品質問題 | 大規模スコープ変更 |
| 極大 | 5 | 2ヶ月超の遅延 | 予算の20%超 | システム利用不可 | プロジェクト中止 |

### 4.2 発生確率の定義

| レベル | スコア | 確率 | 説明 |
|--------|--------|------|------|
| 極低 | 1 | 0-10% | ほぼ発生しない |
| 低 | 2 | 11-30% | 発生の可能性は低い |
| 中 | 3 | 31-50% | 発生する可能性がある |
| 高 | 4 | 51-70% | 発生する可能性が高い |
| 極高 | 5 | 71-100% | ほぼ確実に発生する |

### 4.3 リスクレベルマトリクス

|  | 影響度1 | 影響度2 | 影響度3 | 影響度4 | 影響度5 |
|--|---------|---------|---------|---------|---------|
| **確率5** | 中(5) | 高(10) | 高(15) | 極高(20) | 極高(25) |
| **確率4** | 低(4) | 中(8) | 高(12) | 高(16) | 極高(20) |
| **確率3** | 低(3) | 中(6) | 中(9) | 高(12) | 高(15) |
| **確率2** | 低(2) | 低(4) | 中(6) | 中(8) | 高(10) |
| **確率1** | 低(1) | 低(2) | 低(3) | 低(4) | 中(5) |

### 4.4 リスクレベル別対応方針

| リスクレベル | スコア範囲 | 対応方針 | レビュー頻度 | 報告先 |
|-------------|-----------|----------|-------------|--------|
| 極高 | 20-25 | 即座に対応策実施、プロジェクト計画の見直し | 日次 | SC |
| 高 | 12-19 | 1週間以内に対応策を策定・実行 | 週次 | PM, SC |
| 中 | 6-11 | 対応策を策定し、月次レビューで進捗確認 | 月次 | PM |
| 低 | 1-5 | 受容可能、モニタリング継続 | 月次 | PMO |

## 6. リスク対応計画

### 6.1 対応戦略の定義

| 戦略 | 説明 | 適用場面 |
|------|------|----------|
| 回避（Avoid） | リスクの原因を除去する | リスクが極めて高く、対応コストが合理的な場合 |
| 軽減（Mitigate） | リスクの影響度または発生確率を低下させる | 対応策が明確で実施可能な場合 |
| 移転（Transfer） | リスクを第三者に移転する | 保険、契約によるリスク移転が可能な場合 |
| 受容（Accept） | リスクを認識した上で受け入れる | 影響が小さい、または対応コストが大きすぎる場合 |
| エスカレーション（Escalate） | 上位の意思決定者に判断を委ねる | プロジェクト権限を超える場合 |

## 7. リスク監視・統制

### 7.1 リスクレビュー体制

| 活動 | 頻度 | 参加者 | 内容 |
|------|------|--------|------|
| デイリーリスクチェック | 日次 | 開発チーム | 新規リスクの識別（スクラムの中で実施） |
| 週次リスクレビュー | 週次 | PM、TL、PMO | リスク登録簿の更新、対応状況確認 |
| 月次リスク報告 | 月次 | PM → SC | リスクダッシュボード、Top5リスクの報告 |
| フェーズゲートリスク評価 | フェーズ毎 | SC | フェーズ完了時の残存リスク評価、次フェーズのリスク識別 |

### 7.2 リスクKPI

| KPI | 定義 | 目標値 | 測定頻度 |
|-----|------|--------|----------|
| リスク識別数 | 新規識別リスク数/月 | 安定的に減少傾向 | 月次 |
| リスク対応完了率 | 対応完了リスク数/対応計画リスク数 | 80%以上 | 月次 |
| 残存リスクスコア合計 | 全残存リスクのスコア合計 | 前月比で改善 | 月次 |
| リスク顕在化率 | 顕在化リスク数/識別リスク数 | 20%以下 | 月次 |
| 対応策効果率 | 対応策によりスコア低下したリスク数/対応リスク数 | 70%以上 | 四半期 |

### 7.3 エスカレーションルール

| 条件 | エスカレーション先 | 対応期限 |
|------|-------------------|----------|
| リスクスコア20以上のリスク発生 | SC（緊急報告） | 24時間以内 |
| リスクスコア12-19のリスク発生 | PM → SCへ週次報告 | 1週間以内に対応策策定 |
| 対応策が期限内に完了しない | PM → SC | 次回SC会議で報告 |
| 予算超過10%以上の見込み | PM → SC（緊急報告） | 48時間以内 |
| クリティカルパス上のタスク遅延2週間以上 | PM → SC（緊急報告） | 48時間以内 |
