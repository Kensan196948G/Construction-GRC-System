## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemの品質を保証するためのテスト計画を定める。テスト戦略、テストレベル、テスト環境、テストスケジュール、合格基準を明確にし、体系的なテスト活動を実施する。

### 1.2 対象範囲

| 対象 | 説明 |
|------|------|
| リスク管理（RSK） | リスクの登録・評価・モニタリング・レポート |
| コンプライアンス管理（CMP） | 法令・規制の管理・遵守状況追跡 |
| 統制管理（CTL） | 内部統制の設計・運用・評価 |
| 監査管理（AUD） | 監査計画・実施・報告・フォローアップ |
| レポート管理（RPT） | ダッシュボード・帳票生成・データエクスポート |
| 認証・認可 | ユーザー管理・RBAC・JWT認証 |
| 通知機能 | メール・アプリ内通知・期限アラート |

### 1.3 対象外

- モバイルアプリケーション（将来対応予定）
- 外部システムとのリアルタイム連携（Phase 2以降）
- ハードウェアテスト

## 2. テスト戦略

### 2.1 テストピラミッド

```
          ┌────────────┐
          │   E2E      │  少数・高コスト・低速
          │  テスト     │  主要業務フローのみ
          ├────────────┤
          │  結合       │  中程度
          │  テスト     │  API・DB統合テスト
          ├────────────┤
          │            │  大量・低コスト・高速
          │  単体テスト  │  ビジネスロジック中心
          │            │
          └────────────┘
```

### 2.2 テスト比率の目標

| テストレベル | 比率 | テスト数目安 |
|------------|------|------------|
| 単体テスト | 70% | 2,000件以上 |
| 結合テスト | 20% | 500件以上 |
| E2Eテスト | 10% | 100件以上 |

### 2.3 テストアプローチ

| アプローチ | 適用場面 |
|-----------|---------|
| ブラックボックステスト | 機能テスト、受入テスト |
| ホワイトボックステスト | 単体テスト、コードカバレッジ |
| リスクベーステスト | テストケースの優先度付け |
| 回帰テスト | 変更後の既存機能確認（CI自動実行） |
| 探索的テスト | 受入テスト前のアドホック確認 |

## 3. テストレベル

### 3.1 単体テスト（Unit Test）

| 項目 | 内容 |
|------|------|
| 目的 | 個々のモジュール・関数の正確性を検証する |
| 対象 | モデル、シリアライザー、サービス、ユーティリティ、Vueコンポーネント、Composable |
| ツール | pytest（バックエンド）、Vitest（フロントエンド） |
| 実行タイミング | 開発中（常時）、PRマージ前（CI自動） |
| 担当 | 開発者 |
| カバレッジ目標 | バックエンド: 80%以上、フロントエンド: 70%以上 |

#### テスト対象の詳細

| レイヤー | テスト内容 | 例 |
|---------|----------|-----|
| モデル | バリデーション、プロパティ計算、マネージャー | リスクスコアの計算ロジック |
| シリアライザー | 入出力バリデーション、データ変換 | 日付フォーマット、必須項目チェック |
| サービス | ビジネスロジック | リスク評価アルゴリズム |
| ビュー | レスポンスステータス、権限チェック | 認可エラー時の401/403 |
| コンポーネント | レンダリング、イベント処理、Props | フォームバリデーション表示 |
| ストア | 状態変更、アクション実行 | データ取得・更新処理 |

### 3.2 結合テスト（Integration Test）

| 項目 | 内容 |
|------|------|
| 目的 | モジュール間の連携・統合が正しく機能することを検証する |
| 対象 | API エンドポイント、データベース操作、Celeryタスク、外部サービス連携 |
| ツール | pytest + DRF APIClient、TestContainers |
| 実行タイミング | PRマージ前（CI自動）、開発中（必要時） |
| 担当 | 開発者 |

#### テスト対象の詳細

| テスト種別 | 内容 | 例 |
|-----------|------|-----|
| API結合テスト | エンドポイントの入出力・DB操作 | リスク作成→DB保存→レスポンス確認 |
| DB結合テスト | マイグレーション、複雑なクエリ | 階層データの再帰取得 |
| タスク結合テスト | Celeryタスクの実行・結果 | レポート生成タスクの完了確認 |
| 認証結合テスト | JWT取得→API呼び出し→権限検証 | ロール別アクセス制御 |
| 通知結合テスト | イベント発火→通知作成→配信 | 期限切れアラートの送信 |

### 3.3 システムテスト（System Test）

| 項目 | 内容 |
|------|------|
| 目的 | システム全体が要件を満たすことを検証する |
| 対象 | システム全体（フロントエンド + バックエンド + インフラ） |
| ツール | Playwright（E2E）、手動テスト |
| 実行タイミング | ステージング環境デプロイ後 |
| 担当 | QAチーム |

#### テスト観点

| 観点 | 内容 |
|------|------|
| 機能テスト | 要件定義に基づく全機能の動作確認 |
| 画面遷移テスト | 正常系・異常系の画面遷移フロー |
| データ整合性テスト | 登録・更新・削除後のデータ一貫性 |
| クロスブラウザテスト | Chrome、Firefox、Edge、Safari |
| レスポンシブテスト | デスクトップ、タブレット表示 |
| 国際化テスト | 日本語表示の正常性 |

### 3.4 受入テスト（Acceptance Test）

| 項目 | 内容 |
|------|------|
| 目的 | ビジネス要件を満たし、ユーザーが受け入れ可能であることを確認する |
| 対象 | 主要業務シナリオ |
| ツール | 手動テスト、チェックリスト |
| 実行タイミング | リリース前（ステージング環境） |
| 担当 | プロダクトオーナー、業務担当者 |

#### 受入テストシナリオ

| シナリオID | シナリオ名 | 概要 |
|-----------|----------|------|
| UAT-001 | リスク登録から報告まで | 新規リスク登録→評価→承認→レポート出力 |
| UAT-002 | コンプライアンス遵守確認 | 法令確認→遵守状況登録→期限管理→是正措置 |
| UAT-003 | 内部監査サイクル | 監査計画→実施→指摘事項登録→改善確認 |
| UAT-004 | 統制の有効性評価 | 統制項目確認→テスト実施→評価結果記録 |
| UAT-005 | ダッシュボード活用 | ログイン→ダッシュボード確認→詳細ドリルダウン |
| UAT-006 | ユーザー管理 | ユーザー追加→ロール設定→権限確認 |

## 4. テスト環境

### 4.1 環境構成

| 環境 | 用途 | インフラ | データ |
|------|------|---------|--------|
| ローカル開発 | 単体テスト実行 | Docker Compose | テストDB（自動生成） |
| CI | 自動テスト（単体・結合） | GitHub Actions | テストDB（一時） |
| ステージング | システムテスト・受入テスト | 本番同等構成 | テストデータ（匿名化） |
| パフォーマンステスト | 負荷テスト | 本番同等構成 | 大量テストデータ |

### 4.2 テスト環境の構成要素

| コンポーネント | ステージング環境 |
|--------------|----------------|
| アプリケーションサーバー | Docker コンテナ x 2 |
| データベース | PostgreSQL 16（専用インスタンス） |
| キャッシュ | Redis 7 |
| タスクワーカー | Celery Worker x 2 |
| リバースプロキシ | Nginx |
| メールサーバー | MailHog（テスト用SMTPサーバー） |

### 4.3 テストデータ管理

| 種別 | 管理方法 |
|------|---------|
| マスターデータ | Django fixtures（JSON形式） |
| テスト用ファクトリ | factory_boy（Pythonコードで定義） |
| 大量データ | management command で生成 |
| 匿名化データ | 本番データの匿名化スクリプト |

## 5. テストスケジュール

### 5.1 フェーズ別テスト計画

| フェーズ | 期間 | テスト活動 |
|---------|------|----------|
| Phase 1: 基盤構築 | Week 1-4 | 単体テスト（認証・ユーザー管理） |
| Phase 2: RSK/CMP実装 | Week 5-12 | 単体テスト + 結合テスト（RSK/CMP） |
| Phase 3: CTL/AUD実装 | Week 13-20 | 単体テスト + 結合テスト（CTL/AUD） |
| Phase 4: RPT/統合 | Week 21-26 | 単体テスト + 結合テスト + システムテスト |
| Phase 5: 受入テスト | Week 27-30 | 受入テスト + パフォーマンステスト + セキュリティテスト |
| Phase 6: リリース準備 | Week 31-32 | 回帰テスト + 最終確認 |

### 5.2 継続的テスト実行

| テスト種別 | 実行タイミング | 自動/手動 |
|-----------|--------------|----------|
| 単体テスト | コミット時、PR作成時 | 自動（CI） |
| 結合テスト | PR作成時、マージ時 | 自動（CI） |
| E2Eテスト | ステージングデプロイ時 | 自動（CI） |
| システムテスト | スプリント終了時 | 手動 + 自動 |
| 受入テスト | リリース前 | 手動 |
| パフォーマンステスト | リリース前、月次 | 自動 + 手動 |
| セキュリティテスト | リリース前、週次（スキャン） | 自動 + 手動 |
| 回帰テスト | 全テスト | 自動（CI） |

## 6. 合格基準

### 6.1 テストレベル別合格基準

| テストレベル | 合格基準 |
|------------|---------|
| 単体テスト | テスト通過率: 100%、カバレッジ: Backend 80%以上 / Frontend 70%以上 |
| 結合テスト | テスト通過率: 100%、全APIエンドポイントのテスト完了 |
| システムテスト | 重大バグ: 0件、高バグ: 0件、中バグ: 5件以下（全件対応計画あり） |
| 受入テスト | 全受入テストシナリオ通過、PO承認取得 |
| パフォーマンステスト | 全性能指標がSLA基準を満たすこと |
| セキュリティテスト | CRITICAL/HIGH脆弱性: 0件 |

### 6.2 リリース判定基準

| 項目 | 基準 |
|------|------|
| テスト通過率 | 100%（全テストレベル） |
| 未解決バグ | CRITICAL: 0、HIGH: 0、MEDIUM: 対応計画策定済 |
| カバレッジ | 目標値以上 |
| パフォーマンス | SLA基準クリア |
| セキュリティ | CRITICAL/HIGH脆弱性: 0 |
| ドキュメント | テスト報告書・リリースノート完成 |
| 承認 | QAリード + POの承認 |

## 7. バグ管理

### 7.1 バグの重要度分類

| 重要度 | 定義 | 対応期限 |
|--------|------|---------|
| CRITICAL | システム停止、データ損失、セキュリティ脆弱性 | 即時対応（24時間以内） |
| HIGH | 主要機能が使用不可、回避手段なし | 3営業日以内 |
| MEDIUM | 機能の一部に問題、回避手段あり | 次スプリント |
| LOW | 軽微なUI問題、改善要望 | バックログ管理 |

### 7.2 バグレポートテンプレート

```markdown
## バグタイトル
<!-- 簡潔に問題を記述 -->

## 重要度
<!-- CRITICAL / HIGH / MEDIUM / LOW -->

## 再現手順
1.
2.
3.

## 期待される動作
<!-- 正しい動作 -->

## 実際の動作
<!-- 発生している問題 -->

## 環境情報
- OS:
- ブラウザ:
- バージョン:

## スクリーンショット/ログ
<!-- 可能であれば添付 -->

## 備考
```

## 9. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| テスト工数の不足 | テストカバレッジの低下 | 自動テストの優先実施、リスクベーステストの適用 |
| テスト環境の不安定 | テスト実行の遅延 | Docker化による環境の再現性確保 |
| テストデータの不足 | テストの網羅性低下 | factory_boyによる効率的なデータ生成 |
| 要件変更によるテスト修正 | テストの再作成コスト | テストの保守性を意識した設計 |
| パフォーマンステスト環境の調達 | 本番同等のテスト不可 | クラウド環境の一時利用 |
