## 1. 概要

### 1.1 目的

本文書は、Construction-GRC-Systemのセキュリティテスト計画を定める。GRCシステムは企業のリスク・コンプライアンス情報という機密性の高いデータを扱うため、高水準のセキュリティ確保が必須である。

### 1.2 対象範囲

| 対象 | 説明 |
|------|------|
| Webアプリケーション | フロントエンド（Vue.js）、バックエンドAPI（Django/DRF） |
| 認証・認可 | JWT認証、RBAC、セッション管理 |
| データベース | PostgreSQL のアクセス制御、暗号化 |
| インフラ | Docker コンテナ、Nginx、Redis |
| 依存パッケージ | Python/Node.js の依存ライブラリ |
| CI/CD | GitHub Actions のセキュリティ |

### 1.3 準拠基準

| 基準 | 説明 |
|------|------|
| OWASP Top 10 (2021) | Webアプリケーションの主要脆弱性 |
| OWASP ASVS v4.0 | アプリケーションセキュリティ検証基準 |
| CWE/SANS Top 25 | 最も危険なソフトウェアエラー |
| 個人情報保護法 | 個人データの適切な管理 |
| ISO 27001 | 情報セキュリティマネジメント |

## 2. OWASP Top 10 対策テスト

### 2.1 A01:2021 – アクセス制御の不備（Broken Access Control）

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |
|---------|----------|----------|---------|--------|
| SEC-A01-001 | 水平権限昇格 | 一般ユーザーAのトークンで、ユーザーBのデータにアクセスを試行 | 403 Forbidden が返される | 手動/Burp Suite |
| SEC-A01-002 | 垂直権限昇格 | 一般ユーザーのトークンで管理者用APIを呼び出す | 403 Forbidden が返される | 手動/Burp Suite |
| SEC-A01-003 | IDOR（安全でない直接オブジェクト参照） | URL内のIDを他ユーザーのリソースIDに変更 | 自分に権限のないリソースにはアクセス不可 | 手動 |
| SEC-A01-004 | 未認証アクセス | 認証トークンなしで保護されたAPIを呼び出す | 401 Unauthorized が返される | 自動テスト |
| SEC-A01-005 | 管理画面アクセス制御 | 一般ユーザーで `/admin/` にアクセス | アクセス拒否される | 手動 |
| SEC-A01-006 | APIメソッド制限 | 許可されていないHTTPメソッド（DELETE等）を送信 | 405 Method Not Allowed が返される | 手動 |
| SEC-A01-007 | 部門間データ分離 | 部門Aのユーザーで部門Bのデータにアクセス | 部門Bのデータは表示されない | 自動テスト |

### 2.2 A02:2021 – 暗号化の失敗（Cryptographic Failures）

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |
|---------|----------|----------|---------|--------|
| SEC-A02-001 | HTTPS強制 | HTTPでアクセスを試行 | HTTPSにリダイレクトされる | curl |
| SEC-A02-002 | TLS設定 | TLSバージョンと暗号スイートを確認 | TLS 1.2以上、弱い暗号スイートが無効 | testssl.sh |
| SEC-A02-003 | パスワードハッシュ | データベース内のパスワードフィールドを確認 | bcrypt/PBKDF2 でハッシュ化されている | DB直接確認 |
| SEC-A02-004 | 機密データの暗号化 | 個人情報関連のカラムを確認 | 必要に応じて暗号化されている | DB直接確認 |
| SEC-A02-005 | API応答の機密情報 | API応答にパスワードハッシュ等が含まれないか確認 | 機密フィールドが除外されている | 手動/自動テスト |
| SEC-A02-006 | Cookie属性 | Set-Cookie ヘッダーを確認 | Secure, HttpOnly, SameSite属性が設定されている | ブラウザ開発者ツール |

### 2.3 A03:2021 – インジェクション（Injection）

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |
|---------|----------|----------|---------|--------|
| SEC-A03-001 | SQLインジェクション | 入力フィールドにSQLインジェクションペイロードを入力 | SQLが実行されず、エラーまたは正常な応答が返される | SQLMap/Burp Suite |
| SEC-A03-002 | NoSQLインジェクション | JSONフィールドにNoSQLペイロードを入力 | 意図しないクエリが実行されない | 手動 |
| SEC-A03-003 | XSS（反射型） | URLパラメータにスクリプトを含める | スクリプトが実行されない | OWASP ZAP |
| SEC-A03-004 | XSS（格納型） | 入力フィールドにスクリプトを保存し、表示を確認 | HTMLエスケープされて表示される | OWASP ZAP |
| SEC-A03-005 | XSS（DOM型） | DOMベースのXSSペイロードを実行 | スクリプトが実行されない | 手動 |
| SEC-A03-006 | コマンドインジェクション | ファイル名やパラメータにOSコマンドを含める | コマンドが実行されない | 手動 |
| SEC-A03-007 | LDAPインジェクション | 検索フィールドにLDAPペイロードを入力 | 意図しない検索が実行されない | 手動 |
| SEC-A03-008 | パストラバーサル | ファイルパスに `../` を含める | サーバー上の任意のファイルにアクセスできない | 手動/Burp Suite |

### 2.4 A04:2021 – 安全でない設計（Insecure Design）

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |
|---------|----------|----------|---------|--------|
| SEC-A04-001 | レート制限 | ログインAPIに短時間で大量のリクエストを送信 | レート制限が発動し、429 Too Many Requests が返される | 手動/k6 |
| SEC-A04-002 | アカウントロック | 5回連続でログインに失敗 | アカウントが一時ロックされる | 手動 |
| SEC-A04-003 | パスワードポリシー | 弱いパスワードで登録を試行 | パスワードポリシー違反のエラーが表示される | 手動 |
| SEC-A04-004 | CAPTCHA/Bot対策 | 自動化ツールでフォーム送信を繰り返す | Bot対策が機能する | 手動 |
| SEC-A04-005 | ビジネスロジック不備 | 負の値や極端な値でリスクスコアを計算 | バリデーションエラーまたは安全なデフォルト値 | 手動 |

### 2.5 A05:2021 – セキュリティ設定ミス（Security Misconfiguration）

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |
|---------|----------|----------|---------|--------|
| SEC-A05-001 | デバッグモード | 本番環境で `DEBUG=True` でないか確認 | `DEBUG=False` に設定されている | 設定確認 |
| SEC-A05-002 | デフォルト認証情報 | デフォルトの管理者パスワードでログインを試行 | ログインできない | 手動 |
| SEC-A05-003 | 不要なHTTPメソッド | OPTIONS, TRACE等のメソッドを送信 | 不要なメソッドが無効化されている | curl |
| SEC-A05-004 | エラーメッセージ | 意図的にエラーを発生させる | スタックトレースやDB情報が表示されない | 手動 |
| SEC-A05-005 | HTTPセキュリティヘッダー | レスポンスヘッダーを確認 | 以下のヘッダーが設定されている | SecurityHeaders.com |
| SEC-A05-006 | CORS設定 | 異なるオリジンからAPIを呼び出す | 許可されたオリジンのみアクセス可能 | curl |
| SEC-A05-007 | ディレクトリリスティング | 存在するディレクトリURLにアクセス | ファイル一覧が表示されない | curl |

**必須HTTPセキュリティヘッダー**:

| ヘッダー | 設定値 |
|---------|--------|
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `X-XSS-Protection` | `1; mode=block` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `Content-Security-Policy` | 適切なポリシー |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | 適切なポリシー |

### 2.7 A07:2021 – 識別と認証の失敗（Identification and Authentication Failures）

| SEC-A07-002 | セッション固定攻撃 | ログイン前後でセッションIDが変わるか確認 | ログイン後にトークンが再生成される | 手動 |

### 2.9 A09:2021 – セキュリティログとモニタリングの不備

| テストID | テスト項目 | テスト手順 | 期待結果 | ツール |

| SEC-A09-002 | 不正アクセスログ | 権限のないリソースへのアクセスを試行 | アクセス拒否がログに記録される | ログ確認 |

## 3. 脆弱性スキャン

### 3.1 自動スキャン

| ツール | 対象 | 実行頻度 | 担当 |
|--------|------|---------|------|
| OWASP ZAP | Webアプリケーション全体 | リリース前、月次 | セキュリティチーム |
| Trivy | Dockerイメージ、依存パッケージ | CI（毎PR）、日次 | CI自動実行 |
| pip-audit | Python依存パッケージ | CI（毎PR）、週次 | CI自動実行 |
| npm audit | Node.js依存パッケージ | CI（毎PR）、週次 | CI自動実行 |
| CodeQL | ソースコード静的解析 | CI（main向けPR） | CI自動実行 |
| Gitleaks | シークレット検出 | CI（毎PR） | CI自動実行 |
| Snyk | 依存パッケージ・コンテナ | 日次 | CI自動実行 |

### 3.2 OWASP ZAP スキャン手順

```bash
# OWASP ZAP のDockerイメージを使用したフルスキャン
docker run -t owasp/zap2docker-stable zap-full-scan.py \
  -t https://staging.grc-system.example.com \
  -r zap_report.html \
  -z "-config api.key=<api-key>"

# API スキャン（OpenAPI仕様を使用）
docker run -t owasp/zap2docker-stable zap-api-scan.py \
  -t https://staging.grc-system.example.com/api/docs/openapi.json \
  -f openapi \
  -r zap_api_report.html
```

### 3.3 スキャン結果の対応基準

| 脆弱性レベル | 対応期限 | 対応責任者 |
|------------|---------|----------|
| CRITICAL | 即時対応（24時間以内） | セキュリティリード + 開発リード |
| HIGH | 3営業日以内 | 開発チーム |
| MEDIUM | 次スプリントで対応 | 開発チーム |
| LOW | バックログ管理 | 開発チーム |
| INFO | 記録のみ | - |

## 4. ペネトレーションテスト計画

### 4.1 テスト概要

| 項目 | 内容 |
|------|------|
| テスト種別 | グレーボックステスト（認証情報提供、ソースコード非開示） |
| テスト期間 | 2週間 |
| テスト環境 | ステージング環境（本番同等構成） |
| テスト実施者 | 外部セキュリティベンダーまたは社内セキュリティチーム |

### 4.2 テスト範囲

| 対象 | 範囲 |
|------|------|
| 対象URL | https://staging.grc-system.example.com |
| 対象API | /api/v1/* |
| 対象ロール | 管理者、一般ユーザー、閲覧者、未認証ユーザー |
| 除外項目 | DoS攻撃、データ削除を伴うテスト |

### 4.3 テスト項目

| カテゴリ | テスト内容 |
|---------|----------|
| 認証テスト | パスワード攻撃、セッション管理、トークン操作 |
| 認可テスト | 権限昇格、IDOR、機能レベルアクセス制御 |
| 入力バリデーション | インジェクション、XSS、ファイルアップロード |
| ビジネスロジック | ワークフローバイパス、レース条件 |
| 暗号化 | 通信暗号化、データ暗号化、鍵管理 |
| エラー処理 | 情報漏洩、スタックトレース |
| API固有 | マスアサインメント、パラメータ汚染 |

### 4.4 ペネトレーションテストのフロー

```
1. 事前準備
   - テスト環境の確認
   - テストアカウントの準備
   - テスト範囲の最終確認

2. 情報収集
   - ポートスキャン（Nmap）
   - ディレクトリ探索（Gobuster）
   - 技術スタック特定

3. 脆弱性分析
   - 自動スキャン（OWASP ZAP, Nikto）
   - 手動テスト
   - ビジネスロジックテスト

4. 脆弱性の実証
   - 検出された脆弱性のPoC作成
   - 影響度の評価

5. 報告
   - 脆弱性レポートの作成
   - 改善提案
   - 優先度付き対応一覧

6. 再テスト
   - 修正後の再テスト
   - 修正確認レポート
```

## 6. インシデント対応

### 6.1 脆弱性発見時の対応フロー

```
1. 脆弱性の報告
   └→ セキュリティチームに即時報告

2. 影響度評価
   └→ CVSS スコアに基づく重大度判定

3. 対応優先度決定
   ├→ CRITICAL: 即時対応（パッチ適用、一時遮断）
   ├→ HIGH: 3営業日以内
   ├→ MEDIUM: 次スプリント
   └→ LOW: バックログ

4. 修正・検証
   └→ 修正実装 → セキュリティレビュー → テスト → デプロイ

5. 事後対応
   └→ インシデントレポート作成、再発防止策策定
```

### 6.2 脆弱性の重大度分類（CVSS v3.1）

| CVSS スコア | 重大度 | 例 |
|------------|--------|-----|
| 9.0 - 10.0 | CRITICAL | 認証バイパス、リモートコード実行 |
| 7.0 - 8.9 | HIGH | SQLインジェクション、権限昇格 |
| 4.0 - 6.9 | MEDIUM | 格納型XSS、CSRF |
| 0.1 - 3.9 | LOW | 情報漏洩（軽微）、反射型XSS |
| 0.0 | INFO | ベストプラクティス違反 |
