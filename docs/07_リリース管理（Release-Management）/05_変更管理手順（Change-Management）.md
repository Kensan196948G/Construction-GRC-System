## 1. 目的

本文書は、Construction-GRC-Systemに対する全ての変更を管理するための手順を定義する。変更要求プロセス、影響分析、変更諮問委員会（CAB: Change Advisory Board）の運用を通じて、変更に伴うリスクを最小化し、サービス品質を維持することを目的とする。

---

## 2. 適用範囲

### 2.1 対象となる変更

| 変更カテゴリ | 例 |
|------------|---|
| アプリケーション変更 | 機能追加、バグ修正、UI変更、API変更 |
| インフラ変更 | サーバー構成変更、ネットワーク変更、ストレージ拡張 |
| データベース変更 | スキーマ変更、マイグレーション、データ移行 |
| セキュリティ変更 | アクセス権限変更、暗号化方式変更、認証方式変更 |
| GRC関連変更 | 管理策の追加・変更、法令対応変更、監査要件変更 |
| 設定変更 | 環境変数変更、パラメータ変更、閾値変更 |
| 外部連携変更 | API連携先変更、データ連携方式変更 |

### 2.2 対象外

| 項目 | 理由 |
|------|------|
| 開発環境内のみの変更 | 本番サービスに影響しないため |
| ドキュメントのみの変更 | システム動作に影響しないため |
| 自動テストコードのみの変更 | 本番コードに影響しないため |

## 3. 変更の分類

### 3.1 変更タイプ

| タイプ | 説明 | 承認プロセス | リードタイム |
|-------|------|------------|------------|
| 標準変更 | 事前承認済みの定型変更（パッチ適用等） | 事前承認リストに基づき自動承認 | 1営業日 |
| 通常変更 | 計画的な変更（機能追加、改善等） | CABによる審議・承認 | 5〜10営業日 |
| 緊急変更 | セキュリティ脆弱性、重大障害の修正 | 緊急CAB（事後承認可） | 即時〜1営業日 |
| 大規模変更 | アーキテクチャ変更、メジャーバージョンアップ | 拡大CAB + 経営層承認 | 20営業日以上 |

### 3.2 変更リスクレベル

| リスクレベル | 判定基準 | 承認要件 |
|------------|---------|---------|
| Low | 影響範囲が限定的、ロールバック容易 | テックリード承認 |
| Medium | 複数コンポーネントに影響、ロールバック可能 | CAB承認 |
| High | 広範囲に影響、データ移行を伴う | CAB承認 + セキュリティ承認 |
| Critical | 全システムに影響、破壊的変更 | 拡大CAB + 経営層承認 |

### 3.3 リスクレベル判定マトリクス

```
影響度 ＼ 発生確率  |  低い   |  中程度  |  高い
━━━━━━━━━━━━━━━━━━┼━━━━━━━━━┼━━━━━━━━━┼━━━━━━━
広範囲（全ユーザー） |  High   | Critical | Critical
━━━━━━━━━━━━━━━━━━┼━━━━━━━━━┼━━━━━━━━━┼━━━━━━━
中程度（特定機能）   |  Medium |   High   | Critical
━━━━━━━━━━━━━━━━━━┼━━━━━━━━━┼━━━━━━━━━┼━━━━━━━
限定的（軽微）      |   Low   |  Medium  |   High
```

## 6. 変更諮問委員会（CAB）運用

### 6.1 CABの構成

| 役割 | メンバー | 責務 |
|------|---------|------|
| CAB委員長 | CTO/CIO | 最終承認、会議進行 |
| テクニカルアドバイザー | テックリード | 技術的実現性評価、リスク評価 |
| セキュリティアドバイザー | CISO/セキュリティ責任者 | セキュリティリスク評価 |
| GRCアドバイザー | GRC管理者 | 法令・規格準拠性評価 |
| インフラアドバイザー | インフラリード | インフラ影響評価、デプロイリスク評価 |
| ビジネスアドバイザー | PM/プロダクトオーナー | ビジネス優先度判断 |
| QAアドバイザー | QAリード | テスト品質評価 |

### 6.2 CAB定例会議

| 項目 | 内容 |
|------|------|
| 開催頻度 | 週1回（毎週火曜日 10:00〜11:00） |
| 開催形式 | オンライン会議（必要に応じて対面） |
| 議題 | 新規RFC審議、進行中変更の状況報告、完了変更の振り返り |
| 議事録 | 会議後24時間以内に共有 |
| 出席要件 | 委員長 + 3名以上で成立 |

### 6.3 CAB審議プロセス

```
┌──────────────────────────────────────────────────┐
│               CAB審議プロセス                       │
├──────────────────────────────────────────────────┤
│                                                  │
│  1. RFC提出（会議3営業日前まで）                      │
│     └─► 変更マネージャーが事前配布                    │
│                                                  │
│  2. 事前レビュー（各アドバイザー）                     │
│     ├─► 技術的実現性                               │
│     ├─► セキュリティリスク                           │
│     ├─► GRC準拠性                                  │
│     └─► インフラ影響                                │
│                                                  │
│  3. CAB会議                                       │
│     ├─► RFC概要説明（起案者: 5分）                   │
│     ├─► 影響分析報告（各アドバイザー: 各3分）          │
│     ├─► 質疑応答（10分）                            │
│     └─► 判定（委員長）                              │
│                                                  │
│  4. 判定結果                                       │
│     ├── 承認 ──► 実装フェーズへ                     │
│     ├── 条件付承認 ──► 条件対応後に再提出            │
│     ├── 却下 ──► 起案者へ差し戻し                   │
│     └── 保留 ──► 追加情報要求                       │
│                                                  │
│  5. 記録                                          │
│     └─► 議事録 + RFC更新                           │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 6.4 緊急CAB

通常のCABスケジュール外で緊急の変更が必要な場合の手順。

| 項目 | 内容 |
|------|------|
| 発動条件 | セキュリティ脆弱性（Critical/High）、サービス停止障害、法令違反リスク |
| 招集方法 | Slack緊急チャンネル + 電話 |
| 最小出席者 | CAB委員長 + テクニカルアドバイザー + セキュリティアドバイザー |
| 審議時間 | 最大30分 |
| 承認方法 | 口頭承認（事後書面化） |
| 事後対応 | 翌営業日のCAB定例で正式報告・記録 |

### 6.5 CAB判定基準

| 判定 | 条件 |
|------|------|
| 承認 | リスクが許容範囲内、テスト計画が十分、ロールバック計画がある |
| 条件付承認 | 軽微な修正が必要だが、条件対応後に再審議不要 |
| 却下 | リスクが許容範囲外、代替案が必要、準備不足 |
| 保留 | 追加の影響分析が必要、他の変更との依存関係の確認が必要 |

## 7. 標準変更カタログ

事前承認済みの定型変更は、CAB審議を省略して実施できる。

### 7.1 標準変更一覧

| ID | 変更名 | 説明 | リスク | 実施条件 |
|----|--------|------|--------|---------|
| SC-001 | セキュリティパッチ適用 | OS・ミドルウェアのセキュリティパッチ | Low | テスト合格、ロールバック手順確認済み |
| SC-002 | 依存パッケージ更新（パッチ） | npm/pip パッケージのパッチバージョン更新 | Low | 脆弱性スキャン合格、テスト合格 |
| SC-003 | SSL証明書更新 | SSL/TLS証明書の更新 | Low | 証明書の有効性確認済み |
| SC-004 | ログローテーション設定変更 | ログファイルのローテーション条件変更 | Low | 設定値の妥当性確認済み |
| SC-005 | 監視閾値変更 | アラート閾値の調整 | Low | 過去のメトリクス分析に基づく |
| SC-006 | バックアップスケジュール変更 | バックアップ実行時間の調整 | Low | メンテナンスウィンドウ内 |
| SC-007 | ユーザーロール割当変更 | 既存ロールの割当変更 | Low | GRC管理者承認済み |

### 7.2 標準変更の実施条件

- 標準変更カタログに登録されていること
- 記載された実施条件を全て満たすこと
- 実施記録を残すこと（自動記録を含む）
- 問題発生時は即座にCABに報告すること

## 8. 変更管理メトリクス

### 8.1 KPI

| メトリクス | 目標値 | 測定頻度 |
|-----------|--------|---------|
| 変更成功率 | 95%以上 | 月次 |
| 変更起因障害率 | 5%以下 | 月次 |

| 緊急変更比率 | 全変更の10%以下 | 月次 |
| CAB審議却下率 | 20%以下 | 月次 |

### 8.2 月次レポート項目

| レポート項目 | 内容 |
|------------|------|
| 変更件数 | タイプ別・リスクレベル別の変更件数 |
| 変更成功/失敗件数 | 成功・失敗・ロールバック件数 |

| リードタイム分析 | 起案から完了までの平均・最大時間 |
| CAB審議結果 | 承認・却下・保留の件数 |
| 改善提案 | プロセス改善の提案事項 |
