## 1. 目的

本文書は、建設業統合リスク＆コンプライアンス管理システム（Construction-GRC-System）のリリース計画を定義する。リリーススケジュール、バージョニング戦略、リリースサイクルを明確化し、品質を維持しながら計画的なリリースを実現することを目的とする。

---

## 2. 適用範囲

- Construction-GRC-Systemの全コンポーネント（バックエンド、フロントエンド、インフラ）
- 開発環境（Development）、ステージング環境（Staging）、本番環境（Production）
- 全てのリリースタイプ（メジャー、マイナー、パッチ、ホットフィックス）

## 3. バージョニング戦略（Semantic Versioning）

### 3.1 SemVer規則

本システムはSemantic Versioning 2.0.0（SemVer）を採用する。

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]
```

| バージョン要素 | 意味 | 変更条件 |
|---------------|------|---------|
| MAJOR | メジャーバージョン | 後方互換性のないAPI変更、データベーススキーマの破壊的変更 |
| MINOR | マイナーバージョン | 後方互換性を維持した機能追加、新しいGRC管理策対応 |
| PATCH | パッチバージョン | 後方互換性を維持したバグ修正、セキュリティパッチ |
| PRERELEASE | プレリリース | alpha, beta, rc（リリース候補）|
| BUILD | ビルドメタデータ | ビルド番号、コミットハッシュ |

### 3.2 バージョン例

| バージョン | 説明 |
|-----------|------|
| `1.0.0` | 初回正式リリース |
| `1.1.0` | ISO27001新管理策対応追加 |
| `1.1.1` | リスク評価計算ロジックのバグ修正 |
| `2.0.0` | データモデル大幅改修（破壊的変更） |
| `1.2.0-alpha.1` | 品確法対応機能のアルファ版 |
| `1.2.0-beta.1` | 品確法対応機能のベータ版 |
| `1.2.0-rc.1` | 品確法対応機能のリリース候補 |

### 3.3 ブランチ戦略

```
main ─────────────────────────────────────────────► 本番リリース
  │
  ├── release/1.0.0 ──────────► タグ: v1.0.0
  │
  ├── release/1.1.0 ──────────► タグ: v1.1.0
  │
  ├── develop ────────────────────────────────────► 開発統合ブランチ
  │     │
  │     ├── feature/GRC-101-risk-assessment
  │     ├── feature/GRC-102-compliance-dashboard
  │     └── feature/GRC-103-audit-management
  │
  └── hotfix/1.0.1 ──────────► タグ: v1.0.1
```

| ブランチ | 用途 | 命名規則 |
|---------|------|---------|
| `main` | 本番リリースブランチ | 固定名 |
| `develop` | 開発統合ブランチ | 固定名 |
| `feature/*` | 機能開発ブランチ | `feature/{Issue番号}-{概要}` |
| `release/*` | リリース準備ブランチ | `release/{バージョン}` |
| `hotfix/*` | 緊急修正ブランチ | `hotfix/{バージョン}` |
| `bugfix/*` | バグ修正ブランチ | `bugfix/{Issue番号}-{概要}` |

## 4. リリーススケジュール

### 4.1 リリースサイクル

| リリースタイプ | サイクル | 説明 |
|---------------|---------|------|
| メジャーリリース | 年1〜2回 | 大規模機能追加、法令改正対応、アーキテクチャ変更 |
| マイナーリリース | 月1回（毎月第3水曜日） | 新機能追加、既存機能改善 |
| パッチリリース | 随時（最大週1回） | バグ修正、軽微な改善 |
| ホットフィックス | 緊急時即時 | セキュリティ脆弱性、重大バグ修正 |

### 4.2 年間リリースロードマップ（2026年度）

| リリース | 予定日 | バージョン | 主な内容 |
|---------|--------|-----------|---------|
| Phase 1 | 2026-04-16 | v1.0.0 | 初回リリース：リスク管理基本機能、ISO27001基本管理策対応 |
| Phase 1.1 | 2026-05-21 | v1.1.0 | コンプライアンスダッシュボード、建設業法基本対応 |
| Phase 1.2 | 2026-06-18 | v1.2.0 | 監査管理機能、品確法基本対応 |
| Phase 1.3 | 2026-07-16 | v1.3.0 | 労安法対応、安全衛生管理機能 |
| Phase 2 | 2026-09-17 | v2.0.0 | NIST CSF 2.0完全対応、AI支援リスク分析 |
| Phase 2.1 | 2026-10-15 | v2.1.0 | 高度分析ダッシュボード、レポート自動生成 |
| Phase 2.2 | 2026-11-19 | v2.2.0 | 外部システム連携（経審システム、CCUS等） |
| Phase 3 | 2027-01-21 | v3.0.0 | マルチテナント対応、大規模拡張 |

### 4.3 リリースタイムライン（各マイナーリリース）

```
Day 1-14    : 機能開発期間（feature ブランチ）
Day 15      : フィーチャーフリーズ（develop → release ブランチ作成）
Day 15-18   : 統合テスト・リグレッションテスト
Day 19-20   : ステージング環境デプロイ・UAT（ユーザー受入テスト）
Day 21      : リリース判定会議
Day 21      : 本番リリース（第3水曜日）
Day 22-23   : リリース後監視（ポストリリースモニタリング）
```

## 5. リリースプロセス

### 5.1 リリースフロー概要

```
┌──────────────────────────────────────────────────────────────┐
│                    リリースプロセス全体フロー                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. フィーチャーフリーズ                                        │
│     └─► develop から release/x.y.z ブランチ作成               │
│                                                              │
│  2. リリース準備                                               │
│     ├─► バージョン番号更新                                      │
│     ├─► CHANGELOG 更新                                        │
│     ├─► リリースノート作成                                      │
│     └─► 依存パッケージ最終確認                                   │
│                                                              │
│  3. 品質確認                                                   │
│     ├─► 自動テスト実行（単体・統合・E2E）                         │
│     ├─► セキュリティスキャン                                     │
│     ├─► パフォーマンステスト                                     │
│     ├─► GRC管理策準拠性確認                                     │
│     └─► コンプライアンスチェック                                  │
│                                                              │
│  4. ステージングデプロイ                                         │
│     ├─► ステージング環境展開                                     │
│     ├─► UAT実施                                               │
│     └─► 法令準拠性最終確認                                      │
│                                                              │
│  5. リリース承認                                                │
│     ├─► リリース判定会議                                        │
│     ├─► CAB（変更諮問委員会）承認                                │
│     └─► セキュリティ責任者承認                                   │
│                                                              │
│  6. 本番リリース                                                │
│     ├─► release → main マージ                                 │
│     ├─► タグ付与（vx.y.z）                                     │
│     ├─► 本番環境デプロイ                                        │
│     └─► release → develop マージ（差分統合）                     │
│                                                              │
│  7. リリース後                                                  │
│     ├─► ポストリリースモニタリング（48時間）                       │
│     ├─► リリース報告書作成                                       │
│     └─► 教訓・改善事項記録                                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 ホットフィックスプロセス

ホットフィックスは通常のリリースサイクルとは独立して実施する。

```
本番障害検知
  │
  ├─► 障害レベル判定
  │    ├── Critical: 即時対応（2時間以内）
  │    ├── High: 24時間以内対応
  │    └── Medium以下: 次回パッチリリースで対応
  │
  ├─► hotfix ブランチ作成（main から分岐）
  │
  ├─► 修正実装・テスト
  │
  ├─► 緊急レビュー（最低1名）
  │
  ├─► main マージ・タグ付与
  │
  ├─► 本番デプロイ
  │
  └─► develop へバックマージ
```

## 6. リリース環境

### 6.1 環境一覧

| 環境 | 用途 | デプロイ頻度 | URL パターン |
|------|------|------------|-------------|
| Development | 開発・デバッグ | 随時（CI/CD自動） | `dev.grc.example.co.jp` |
| Staging | 統合テスト・UAT | リリース前（手動承認） | `stg.grc.example.co.jp` |
| Production | 本番運用 | リリーススケジュールに準拠 | `grc.example.co.jp` |

### 6.2 環境別設定

| 設定項目 | Development | Staging | Production |
|---------|-------------|---------|------------|
| データベース | PostgreSQL (ローカル) | PostgreSQL (RDS) | PostgreSQL (RDS Multi-AZ) |
| ログレベル | DEBUG | INFO | WARN |
| SSL/TLS | 自己署名証明書 | Let's Encrypt | 商用証明書 |
| バックアップ | なし | 日次 | リアルタイムレプリケーション + 日次スナップショット |
| 監視 | 基本メトリクス | 全メトリクス | 全メトリクス + アラート |
| 認証 | ローカル認証 | SAML/OIDC | SAML/OIDC + MFA必須 |

## 7. リリース判定基準

### 7.1 リリース可否判定マトリクス

| 判定項目 | 合格基準 | 必須/推奨 |
|---------|---------|----------|
| 自動テストカバレッジ | 80%以上 | 必須 |
| 全自動テスト合格 | 100% | 必須 |
| セキュリティスキャン | Critical/High脆弱性 0件 | 必須 |
| パフォーマンステスト | レスポンス p95 < 3秒 | 必須 |
| GRC管理策適合性 | 対象管理策100%適合 | 必須 |
| UAT合格 | 主要シナリオ100%合格 | 必須 |
| リリースノート作成 | 完了 | 必須 |
| ロールバック手順確認 | 完了 | 必須 |
| 法令準拠性確認 | 全対象法令確認完了 | 必須 |
| 既知の問題文書化 | 完了 | 推奨 |
| 運用マニュアル更新 | 完了 | 推奨 |

### 7.2 リリースブロッカー

以下のいずれかに該当する場合、リリースをブロックする。

| ブロッカー条件 | 対応方針 |
|-------------|---------|
| Critical/Highバグ未解決 | バグ修正完了まで延期 |
| セキュリティ脆弱性（Critical） | 脆弱性修正完了まで延期 |
| GRC管理策不適合 | 適合確認完了まで延期 |
| データマイグレーション失敗 | マイグレーション修正完了まで延期 |
| 法令準拠性未確認 | 確認完了まで延期 |
| 承認者不在 | 承認完了まで延期 |
